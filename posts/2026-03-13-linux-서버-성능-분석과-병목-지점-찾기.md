# Linux 서버 성능 분석과 병목 지점 찾기

## 개요

프로덕션 환경에서 서버가 갑자기 느려지거나, 특정 시간대에 응답 지연이 발생하는 경험은 백엔드 개발자라면 누구나 겪게 된다. 이때 "왜 느린가?"라는 질문에 답하려면 체계적인 성능 분석 접근법이 필요하다.

Linux 서버 성능 문제는 대부분 네 가지 영역 중 하나 혹은 복합적으로 발생한다. **CPU**, **메모리**, **디스크 I/O**, **네트워크**가 그것이다. 그리고 각 영역마다 적절한 진단 도구와 해석 방법이 다르다.

이 글에서는 실무에서 바로 활용할 수 있는 성능 분석 워크플로우를 소개하고, 각 병목 지점을 찾는 구체적인 명령어와 스크립트를 다룬다.

---

## 핵심 개념: USE 방법론

성능 분석에 접근할 때 가장 효과적인 프레임워크 중 하나는 Brendan Gregg가 제안한 **USE 방법론(Utilization, Saturation, Errors)**이다.

- **Utilization**: 리소스가 얼마나 사용되고 있는가 (예: CPU 70%)
- **Saturation**: 리소스가 처리할 수 없어서 대기 중인 작업이 있는가 (예: Run Queue 길이)
- **Errors**: 에러가 발생하고 있는가 (예: 디스크 I/O 에러, 패킷 드롭)

이 세 가지 관점으로 각 서브시스템을 점검하면 병목 지점을 빠르게 좁힐 수 있다.

## 실전 예제

### 1단계: 전체 현황 파악 — `top` / `htop` / `glances`

가장 먼저 서버 전체 상태를 빠르게 확인한다.

```bash
# 실시간 프로세스 모니터링
top -d 1

# 더 직관적인 UI (설치 필요)
htop

# 종합 대시보드 (Python 기반)
glances
```

`top` 출력에서 가장 먼저 봐야 할 항목은 다음과 같다.

```
top - 14:23:01 up 10 days,  3:42,  2 users,  load average: 4.12, 3.87, 3.50
Tasks: 320 total,   3 running, 317 sleeping
%Cpu(s): 78.3 us,  5.2 sy,  0.0 ni, 12.1 id,  3.8 wa,  0.0 hi,  0.6 si
MiB Mem :  15872.0 total,    512.3 free,  12048.1 used,   3311.6 buff/cache
```

- **load average**: CPU 코어 수와 비교하여 1.0을 초과하면 포화 상태
- **wa (I/O wait)**: 5% 이상이면 디스크 병목 의심
- **us (user space)**: 애플리케이션 레벨 CPU 사용률

### 2단계: CPU 병목 분석

```bash
# CPU 코어별 사용률 확인
mpstat -P ALL 1 5

# 어떤 프로세스가 CPU를 많이 쓰는지
pidstat -u 1 10

# CPU 컨텍스트 스위칭 확인
vmstat 1 10
```

`vmstat` 출력 예시:

```
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 5  0      0 524288  81920 3112960    0    0     0   128 4523 12847 78  6 12  4  0
```

- **r (run queue)**: CPU 코어 수보다 지속적으로 크면 CPU 포화
- **cs (context switches)**: 과도하게 높으면 스레드 경합 의심
- **in (interrupts)**: 높은 인터럽트는 네트워크 트래픽 폭주 가능성

**CPU 병목 심화 분석 — perf 사용:**

```bash
# 시스템 전체 CPU 샘플링 (10초)
perf top -d 10

# 특정 프로세스 프로파일링 (PID: 1234)
perf record -g -p 1234 sleep 30
perf report --stdio

# 플레임 그래프 생성 (FlameGraph 도구 필요)
perf record -F 99 -p 1234 -g -- sleep 30
perf script | ./stackcollapse-perf.pl | ./flamegraph.pl > flame.svg
```

### 3단계: 메모리 병목 분석

```bash
# 메모리 상세 현황
free -h
cat /proc/meminfo

# 프로세스별 메모리 사용
ps aux --sort=-%mem | head -20

# 메모리 누수 의심 시 — smem 활용
smem -r -s rss | head -20

# OOM Killer 발동 이력 확인
dmesg | grep -i "out of memory"
journalctl -k | grep -i oom
```

**스왑 사용 여부 확인:**

```bash
# 스왑 사용 중이면 심각한 메모리 부족 신호
swapon --show
vmstat 1 5  # si/so 컬럼 확인 (swap in/out)
```

`si`, `so` 값이 0이 아니라면 메모리가 부족하여 디스크로 스왑이 발생하고 있다는 뜻이다. 이 상태가 지속되면 I/O 병목까지 연쇄적으로 유발된다.

### 4단계: 디스크 I/O 병목 분석

```bash
# 디스크 I/O 실시간 모니터링
iostat -x 1 10

# 프로세스별 I/O 사용량
iotop -o -d 1

# 디스크 레이턴시 확인
ioping -c 10 /var/data
```

`iostat -x` 출력에서 핵심 지표:

```
Device  r/s   w/s  rMB/s  wMB/s  await  r_await  w_await  %util
sda    12.3  85.7   0.98  10.24  124.3    8.2     138.7    97.2
```

- **%util**: 90% 이상이면 디스크 포화 상태
- **await**: I/O 요청 평균 대기 시간 (ms), 높을수록 병목
- **w_await**: 쓰기 대기 시간이 높다면 DB 워크로드 집중 의심

**디스크 I/O 병목 원인 추적:**

```bash
# 어떤 파일이 I/O를 많이 유발하는지
lsof | awk '{print $9}' | sort | uniq -c | sort -rn | head -20

# 파일시스템 레벨 통계
df -h
du -sh /var/log/* | sort -rh | head -10

# I/O 스케줄러 확인
cat /sys/block/sda/queue/scheduler
```

### 5단계: 네트워크 병목 분석

```bash
# 네트워크 인터페이스 통계
sar -n DEV 1 10

# 실시간 네트워크 트래픽
nload eth0
iftop -i eth0

# TCP 연결 상태 분석
ss -s
ss -tnp | awk '{print $1}' | sort | uniq -c | sort -rn

# TIME_WAIT 과다 확인
ss -tan | grep TIME_WAIT | wc -l

# 패킷 드롭 확인
netstat -s | grep -i "failed\|error\|drop\|overflow"
```

**TCP 연결 수 분석 스크립트:**

```bash
#!/bin/bash
# tcp_state_count.sh — TCP 상태별 연결 수 출력

echo "=== TCP Connection State Summary ==="
echo "Timestamp: $(date)"
echo ""
ss -tan | awk 'NR>1 {print $1}' | sort | uniq -c | sort -rn | \
  awk '{printf "%-15s : %d\n", $2, $1}'

echo ""
echo "=== Top 10 Remote IPs by Connection Count ==="
ss -tan | awk 'NR>1 {print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn | head -10
```

### 6단계: 종합 진단 스크립트

실무에서 장애 발생 시 빠르게 전체 현황을 캡처하는 스크립트를 만들어두면 유용하다.

```bash
#!/bin/bash
# server_diagnosis.sh — 서버 상태 종합 스냅샷

LOGDIR="/tmp/diagnosis_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$LOGDIR"

echo "[*] 기본 시스템 정보 수집..."
uname -a > "$LOGDIR/system_info.txt"
uptime >> "$LOGDIR/system_info.txt"
date >> "$LOGDIR/system_info.txt"

echo "[*] CPU 분석..."
mpstat -P ALL 1 5 > "$LOGDIR/cpu_mpstat.txt"
vmstat 1 10 > "$LOGDIR/vmstat.txt"
ps aux --sort=-%cpu | head -30 > "$LOGDIR/top_cpu_procs.txt"

echo "[*] 메모리 분석..."
free -h > "$LOGDIR/memory.txt"
cat /proc/meminfo >> "$LOGDIR/memory.txt"
ps aux --sort=-%mem | head -30 > "$LOGDIR/top_mem_procs.txt"

echo "[*] 디스크 I/O 분석..."
iostat -x 1 5 > "$LOGDIR/iostat.txt"
df -h > "$LOGDIR/disk_usage.txt"

echo "[*] 네트워크 분석..."
ss -s > "$LOGDIR/network_summary.txt"
ss -tan | awk '{print $1}' | sort | uniq -c >> "$LOGDIR/network_summary.txt"
netstat -s 2>/dev/null | grep -i "error\|drop\|fail" > "$LOGDIR/network_errors.txt"

echo "[*] 시스템 로그 수집..."
dmesg | tail -100 > "$LOGDIR/dmesg.txt"
journalctl -n 200 --no-pager > "$LOGDIR/journal.txt" 2>/dev/null

# 압축 아카이브 생성
tar -czf "${LOGDIR}.tar.gz" "$LOGDIR"
echo "[+] 진단 완료: ${LOGDIR}.tar.gz"
```

## 주의사항 및 트레이드오프

### 관찰 자체가 부하를 유발할 수 있다

`perf`, `strace`, `tcpdump` 같은 심층 분석 도구는 그 자체로 CPU와 I/O 부하를 유발한다. 프로덕션 환경에서 사용할 때는 샘플링 주기와 캡처 범위를 최소화하고, 가능하면 스테이징 환경에서 먼저 검증한다.

```bash
# strace는 시스템콜 추적 오버헤드가 크다 — 반드시 짧게 사용
strace -p 1234 -c -f sleep 5  # 5초만 통계 수집
```

### 평균값보다 퍼센타일에 주목하라

`await` 평균이 10ms라도 P99가 500ms라면 사용자의 1%는 심각한 지연을 경험하고 있다. 가능하면 히스토그램으로 분포를 확인해야 한다.

```bash
# latency 분포 확인 (bpftrace 사용)
bpftrace -e 'tracepoint:block:block_rq_complete
  { @usecs = hist((nsecs - @start[$1]) / 1000); }
  tracepoint:block:block_rq_insert { @start[$1] = nsecs; }'
```

### 점진적 접근 — 하나씩 바꿔가며 검증

병목 지점을 찾았다고 해서 여러 파라미터를 동시에 변경하면 무엇이 효과가 있었는지 알 수 없다. 반드시 한 번에 하나씩 변경하고, 변경 전후의 메트릭을 기록해두어야 한다.

### Kernel Parameter 튜닝의 위험성

`/etc/sysctl.conf`를 통한 커널 파라미터 튜닝은 효과가 크지만, 워크로드 특성을 잘못 파악하면 오히려 역효과가 날 수 있다.

```bash
# 자주 사용되는 네트워크 튜닝 — 무분별한 적용은 금물
net.core.somaxconn = 65535
net.ipv4.tcp_tw_reuse = 1
net.ipv4.ip_local_port_range = 10000 65535
vm.swappiness = 10
```

이런 파라미터들은 반드시 현재 워크로드에서 `ss`, `vmstat` 등으로 현상을 확인한 후 적용해야 한다.

## 정리

Linux 서버 성능 분석은 막연히 "뭔가 느리다"는 현상에서 시작해 USE 방법론에 따라 체계적으로 좁혀가는 과정이다.
