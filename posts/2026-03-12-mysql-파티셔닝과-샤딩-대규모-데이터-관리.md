# MySQL 파티셔닝과 샤딩 대규모 데이터 관리

## 개요

서비스 규모가 커질수록 단일 테이블에 수억 건 이상의 데이터가 쌓이는 상황은 필연적으로 찾아옵니다. 이때 쿼리 성능 저하, 인덱스 비대화, 백업/복구 시간 증가 등 다양한 문제가 수면 위로 올라옵니다. MySQL에서 이를 해결하기 위한 대표적인 전략이 **파티셔닝(Partitioning)** 과 **샤딩(Sharding)** 입니다.

두 개념은 종종 혼용되지만 명확히 다릅니다. 파티셔닝은 **단일 MySQL 인스턴스 내**에서 테이블을 물리적으로 분할하는 기법이고, 샤딩은 **여러 MySQL 인스턴스(노드)** 에 데이터를 분산 저장하는 아키텍처입니다. 이 글에서는 각 개념의 내부 동작 원리부터 실전 적용 전략, 그리고 트레이드오프까지 깊이 있게 다룹니다.

---

## 핵심 개념

### MySQL 파티셔닝

파티셔닝은 하나의 논리적 테이블을 여러 물리적 파티션으로 나누는 기법입니다. MySQL 옵티마이저는 쿼리의 `WHERE` 조건을 분석해 필요한 파티션만 스캔하는 **파티션 프루닝(Partition Pruning)** 을 수행합니다.

MySQL이 지원하는 파티셔닝 유형은 크게 네 가지입니다.

| 유형 | 설명 | 주요 사용 사례 |
|------|------|----------------|
| RANGE | 연속된 범위 기준 분할 | 날짜, 연도별 로그 |
| LIST | 명시적 값 목록 기준 분할 | 지역 코드, 카테고리 |
| HASH | 해시 함수 기준 균등 분할 | 균등 분산이 목표일 때 |
| KEY | MySQL 내장 해시 사용 | PK 기반 균등 분산 |

### 샤딩

샤딩은 데이터를 **수평 분할(Horizontal Partitioning)** 하여 여러 독립적인 데이터베이스 인스턴스에 저장하는 방식입니다. 각 인스턴스를 **샤드(Shard)** 라고 부르며, 어떤 데이터가 어느 샤드에 있는지를 결정하는 기준을 **샤딩 키(Shard Key)** 라고 합니다.

샤딩 전략에는 크게 세 가지가 있습니다.

- **범위 기반(Range-based)**: `user_id 1~100만 → shard-1`, `100만~200만 → shard-2`
- **해시 기반(Hash-based)**: `shard = user_id % shard_count`
- **디렉토리 기반(Directory-based)**: 별도의 라우팅 테이블로 매핑 관리

## 실전 예제

### 1. RANGE 파티셔닝 — 로그 테이블 월별 관리

대용량 로그 테이블의 전형적인 패턴입니다. 월별로 파티션을 나누면 오래된 데이터를 `ALTER TABLE DROP PARTITION` 한 줄로 빠르게 삭제할 수 있습니다.

```sql
CREATE TABLE access_log (
    id         BIGINT       NOT NULL AUTO_INCREMENT,
    user_id    BIGINT       NOT NULL,
    action     VARCHAR(100) NOT NULL,
    created_at DATETIME     NOT NULL,
    PRIMARY KEY (id, created_at)  -- 파티션 키는 PK에 포함되어야 함
) ENGINE=InnoDB
PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
    PARTITION p202401 VALUES LESS THAN (202402),
    PARTITION p202402 VALUES LESS THAN (202403),
    PARTITION p202403 VALUES LESS THAN (202404),
    PARTITION p202404 VALUES LESS THAN (202405),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

파티션 프루닝이 제대로 동작하는지 확인하려면 `EXPLAIN`을 활용합니다.

```sql
-- 파티션 프루닝 확인
EXPLAIN SELECT * FROM access_log
WHERE created_at BETWEEN '2024-02-01' AND '2024-02-28';

-- 결과의 partitions 컬럼에서 p202402만 스캔하는지 확인
-- partitions: p202402
```

오래된 파티션 삭제는 `DELETE` 없이 즉시 처리됩니다.

```sql
-- 1월 데이터 전체 삭제 (메타데이터 조작만으로 즉시 완료)
ALTER TABLE access_log DROP PARTITION p202401;

-- 신규 파티션 추가 (p_future를 재구성)
ALTER TABLE access_log REORGANIZE PARTITION p_future INTO (
    PARTITION p202405 VALUES LESS THAN (202406),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

### 2. LIST 파티셔닝 — 지역별 데이터 분리

```sql
CREATE TABLE orders (
    order_id   BIGINT       NOT NULL AUTO_INCREMENT,
    region_cd  TINYINT      NOT NULL COMMENT '1:서울, 2:경기, 3:부산, 4:기타',
    amount     DECIMAL(15,2),
    ordered_at DATETIME     NOT NULL,
    PRIMARY KEY (order_id, region_cd)
) ENGINE=InnoDB
PARTITION BY LIST (region_cd) (
    PARTITION p_seoul  VALUES IN (1),
    PARTITION p_gyeong VALUES IN (2),
    PARTITION p_busan  VALUES IN (3),
    PARTITION p_others VALUES IN (4, 5, 6)
);
```

### 3. 해시 기반 샤딩 — Spring + MyBatis 라우팅 구현

실무에서 샤딩은 애플리케이션 레벨에서 직접 구현하거나 ProxySQL, Vitess 같은 미들웨어를 활용합니다. 아래는 Spring의 `AbstractRoutingDataSource`를 이용한 동적 샤드 라우팅 예제입니다.

```java
// 샤드 컨텍스트 홀더
public class ShardContextHolder {
    private static final ThreadLocal<Integer> SHARD_KEY = new ThreadLocal<>();

    public static void setShardKey(long userId) {
        SHARD_KEY.set((int) (userId % ShardConfig.SHARD_COUNT));
    }

    public static Integer getShardKey() {
        return SHARD_KEY.get();
    }

    public static void clear() {
        SHARD_KEY.remove();
    }
}
```

```java
// 동적 DataSource 라우팅
@Component
public class ShardRoutingDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        Integer shardIndex = ShardContextHolder.getShardKey();
        if (shardIndex == null) {
            return "shard-0"; // 기본 샤드
        }
        return "shard-" + shardIndex;
    }
}
```

```java
// DataSource 설정
@Configuration
public class ShardDataSourceConfig {

    public static final int SHARD_COUNT = 4;

    @Bean
    public DataSource routingDataSource(
            @Qualifier("shard0DataSource") DataSource shard0,
            @Qualifier("shard1DataSource") DataSource shard1,
            @Qualifier("shard2DataSource") DataSource shard2,
            @Qualifier("shard3DataSource") DataSource shard3) {

        Map<Object, Object> targetDataSources = new HashMap<>();
        targetDataSources.put("shard-0", shard0);
        targetDataSources.put("shard-1", shard1);
        targetDataSources.put("shard-2", shard2);
        targetDataSources.put("shard-3", shard3);

        ShardRoutingDataSource routingDataSource = new ShardRoutingDataSource();
        routingDataSource.setTargetDataSources(targetDataSources);
        routingDataSource.setDefaultTargetDataSource(shard0);
        return routingDataSource;
    }
}
```

```java
// 서비스 레이어에서의 사용
@Service
@RequiredArgsConstructor
public class UserOrderService {

    private final OrderRepository orderRepository;

    public List<Order> getOrdersByUser(long userId) {
        try {
            ShardContextHolder.setShardKey(userId);
            return orderRepository.findByUserId(userId);
        } finally {
            ShardContextHolder.clear(); // 반드시 정리
        }
    }
}
```

### 4. 샤딩 환경에서의 글로벌 ID 생성

샤딩 환경에서는 `AUTO_INCREMENT`가 샤드마다 중복될 수 있으므로, 전역 유일 ID 전략이 필요합니다.

```java
// Snowflake ID 생성기 (간략 구현)
public class SnowflakeIdGenerator {
    private final long machineId;       // 샤드 번호 또는 서버 ID
    private final long epoch = 1700000000000L;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public SnowflakeIdGenerator(long machineId) {
        this.machineId = machineId & 0x3FF; // 10비트
    }

    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & 0xFFF; // 12비트 시퀀스
            if (sequence == 0) {
                while (System.currentTimeMillis() <= lastTimestamp) { /* spin */ }
                timestamp = System.currentTimeMillis();
            }
        } else {
            sequence = 0;
        }
        lastTimestamp = timestamp;
        // [41bit timestamp | 10bit machineId | 12bit sequence]
        return ((timestamp - epoch) << 22) | (machineId << 12) | sequence;
    }
}
```

## 주의사항 및 트레이드오프

### 파티셔닝 주의사항

**1. 파티션 키 설계가 핵심**
파티션 프루닝이 동작하려면 `WHERE` 절에 파티션 키가 반드시 포함되어야 합니다. 파티션 키 없이 조회하면 **풀 파티션 스캔**이 발생하여 오히려 성능이 더 나빠질 수 있습니다.

```sql
-- 파티션 프루닝 미동작 (비효율적)
SELECT * FROM access_log WHERE user_id = 1234;

-- 파티션 프루닝 동작 (효율적)
SELECT * FROM access_log
WHERE user_id = 1234 AND created_at >= '2024-03-01';
```

**2. 고유 인덱스 제약**
MySQL 파티셔닝에서 `UNIQUE INDEX`나 `PRIMARY KEY`에는 반드시 파티션 키가 포함되어야 합니다. 이로 인해 복합 PK 설계가 강제되는 경우가 많아 스키마 설계 시 미리 고려해야 합니다.

**3. 파티션 수 제한**
MySQL 8.0 기준 테이블당 최대 8,192개 파티션을 지원합니다. 일별 파티셔닝 시 약 22년치 분량이므로 일반적으로는 문제없지만, 서브파티셔닝 시 주의가 필요합니다.

### 샤딩 주의사항

**1. 크로스 샤드 쿼리 문제**
여러 샤드에 걸친 `JOIN`이나 집계 쿼리(`COUNT`, `SUM`)는 애플리케이션 레벨에서 각 샤드에 분산 요청 후 병합해야 합니다. 이는 코드 복잡도를 크게 높입니다.

**2. 샤드 재분배(Resharding) 비용**
서비스 성장으로 샤드 수를 늘려야 할 때 기존 데이터의 재분배가 필요합니다. `user_id % 4` → `user_id % 8`로 변경하면 데이터의 절반을 이동해야 합니다. **일관된 해싱(Consistent Hashing)** 을 처음부터 도입하면 재분배 비용을 최소화할 수 있습니다.

**3. 분산 트랜잭션**
샤드 간 데이터 일관성 보장은 매우 어렵습니다. 2PC(Two-Phase Commit)는 성능 오버헤드가 크고, 일반적으로 **Saga 패턴**이나 **이벤트 기반 최종 일관성** 방식을 선택합니다.

**4. 운영 복잡도 폭증**
샤드 수만큼 모니터링, 백업, 스키마 마이그레이션 대상이 늘어납니다. Flyway/Liquibase로 마이그레이션을 자동화하고, 각 샤드에 동일한 스크립트를 순차 적용하는 파이프라인을 구축해야 합니다.
