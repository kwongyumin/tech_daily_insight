# DAO(탈중앙화 자율 조직) 거버넌스 설계 원리

## 개요

블록체인 생태계가 성숙하면서 단순한 토큰 발행이나 NFT를 넘어 **조직 자체를 온체인으로 운영**하는 DAO(Decentralized Autonomous Organization)가 주목받고 있다. Compound, Uniswap, MakerDAO 같은 대형 프로토콜이 DAO 거버넌스를 통해 수십억 달러 규모의 프로토콜을 운영하는 것을 보면, 이제 DAO는 단순한 실험이 아닌 실제 소프트웨어 아키텍처의 한 형태로 자리잡았다.

그러나 DAO를 단순히 "투표 기능이 있는 스마트 컨트랙트" 정도로만 이해하고 설계하면 수많은 함정에 빠지게 된다. 본 포스팅에서는 **DAO 거버넌스 시스템을 설계할 때 반드시 고려해야 할 핵심 원리와 실전 예제**를 다룬다.

---

## 핵심 개념

### 1. 거버넌스 라이프사이클

DAO 거버넌스에서 제안(Proposal)은 일반적으로 다음 단계를 거친다.

```
[제안 생성] → [투표 기간] → [타임락(Timelock)] → [실행]
                               ↓
                          [취소 가능 구간]
```

- **Proposal**: 온체인에서 실행될 트랜잭션의 묶음
- **Voting Period**: 토큰 홀더들이 찬성/반대/기권을 행사하는 기간
- **Timelock**: 투표 통과 후 실제 실행 전까지의 지연 시간 (보안의 핵심)
- **Quorum**: 제안이 유효하려면 충족해야 할 최소 투표 참여량

### 2. 투표 메커니즘의 종류

| 방식 | 설명 | 특징 |
|------|------|------|
| Token-weighted | 보유 토큰 수 비례 투표권 | 가장 일반적, 고래 문제 존재 |
| Quadratic Voting | 투표권이 토큰의 제곱근에 비례 | 평등성 향상, Sybil 공격 취약 |
| Conviction Voting | 시간 경과에 따라 투표 가중치 증가 | 장기 참여 유도 |
| Delegation | 다른 주소에 투표권 위임 | 참여율 개선, Compound 모델 |

### 3. 온체인 vs 오프체인 거버넌스

**온체인 거버넌스**는 Snapshot 없이 모든 투표가 블록체인 위에서 이루어진다. 투명하고 탈중앙화되어 있지만 가스비 부담이 크다. **오프체인 거버넌스**는 Snapshot을 통해 서명 기반으로 투표하고, 실행만 온체인에서 한다. 참여율은 높지만 다중서명(Multisig) 같은 신뢰 레이어가 필요하다.

실무에서는 **오프체인 시그널링 + 온체인 실행**의 하이브리드 모델을 많이 사용한다.

## 실전 예제

OpenZeppelin의 `Governor` 프레임워크를 기반으로 실제 거버넌스 컨트랙트를 설계해보자.

### 거버넌스 토큰 (ERC20Votes)

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Votes.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract GovernanceToken is ERC20Votes, Ownable {
    uint256 public constant MAX_SUPPLY = 100_000_000 * 1e18; // 1억 토큰

    constructor(address initialOwner)
        ERC20("DAO Token", "DAOT")
        ERC20Permit("DAO Token")
        Ownable(initialOwner)
    {
        // 초기 토큰 분배: 팀 20%, 커뮤니티 80%
        _mint(initialOwner, MAX_SUPPLY * 20 / 100);
    }

    function mint(address to, uint256 amount) external onlyOwner {
        require(totalSupply() + amount <= MAX_SUPPLY, "Exceeds max supply");
        _mint(to, amount);
    }

    // 투표권 위임 없이는 투표 불가 - 반드시 delegate() 호출 필요
    // self-delegate: token.delegate(msg.sender)
}
```

> **중요**: ERC20Votes를 사용할 때 토큰 홀더는 반드시 `delegate()`를 호출해야 투표 스냅샷에 포함된다. 이를 UX에서 명확히 안내해야 한다.

### 거버넌스 컨트랙트

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/governance/Governor.sol";
import "@openzeppelin/contracts/governance/extensions/GovernorSettings.sol";
import "@openzeppelin/contracts/governance/extensions/GovernorCountingSimple.sol";
import "@openzeppelin/contracts/governance/extensions/GovernorVotes.sol";
import "@openzeppelin/contracts/governance/extensions/GovernorVotesQuorumFraction.sol";
import "@openzeppelin/contracts/governance/extensions/GovernorTimelockControl.sol";

contract DAOGovernor is
    Governor,
    GovernorSettings,
    GovernorCountingSimple,
    GovernorVotes,
    GovernorVotesQuorumFraction,
    GovernorTimelockControl
{
    constructor(
        IVotes _token,
        TimelockController _timelock
    )
        Governor("DAOGovernor")
        GovernorSettings(
            7200,    // 투표 딜레이: ~1일 (블록 기준)
            50400,   // 투표 기간: ~1주일
            100_000e18 // 제안 임계값: 10만 토큰 보유자만 제안 가능
        )
        GovernorVotes(_token)
        GovernorVotesQuorumFraction(4) // 전체 투표권의 4% 이상 참여 필요
        GovernorTimelockControl(_timelock)
    {}

    // 필수 오버라이드
    function votingDelay()
        public view override(Governor, GovernorSettings)
        returns (uint256)
    {
        return super.votingDelay();
    }

    function votingPeriod()
        public view override(Governor, GovernorSettings)
        returns (uint256)
    {
        return super.votingPeriod();
    }

    function quorum(uint256 blockNumber)
        public view override(Governor, GovernorVotesQuorumFraction)
        returns (uint256)
    {
        return super.quorum(blockNumber);
    }

    function state(uint256 proposalId)
        public view override(Governor, GovernorTimelockControl)
        returns (ProposalState)
    {
        return super.state(proposalId);
    }

    function proposalNeedsQueuing(uint256 proposalId)
        public view override(Governor, GovernorTimelockControl)
        returns (bool)
    {
        return super.proposalNeedsQueuing(proposalId);
    }

    function proposalThreshold()
        public view override(Governor, GovernorSettings)
        returns (uint256)
    {
        return super.proposalThreshold();
    }

    function _queueOperations(
        uint256 proposalId,
        address[] memory targets,
        uint256[] memory values,
        bytes[] memory calldatas,
        bytes32 descriptionHash
    ) internal override(Governor, GovernorTimelockControl) returns (uint48) {
        return super._queueOperations(proposalId, targets, values, calldatas, descriptionHash);
    }

    function _executeOperations(
        uint256 proposalId,
        address[] memory targets,
        uint256[] memory values,
        bytes[] memory calldatas,
        bytes32 descriptionHash
    ) internal override(Governor, GovernorTimelockControl) {
        super._executeOperations(proposalId, targets, values, calldatas, descriptionHash);
    }

    function _cancel(
        address[] memory targets,
        uint256[] memory values,
        bytes[] memory calldatas,
        bytes32 descriptionHash
    ) internal override(Governor, GovernorTimelockControl) returns (uint256) {
        return super._cancel(targets, values, calldatas, descriptionHash);
    }

    function _executor()
        internal view override(Governor, GovernorTimelockControl)
        returns (address)
    {
        return super._executor();
    }
}
```

### 제안 생성 및 투표 예제 (ethers.js)

```javascript
const { ethers } = require("ethers");

async function createAndVoteProposal(governor, targetContract, signer) {
  // 1. 실행할 트랜잭션 인코딩
  const transferCalldata = targetContract.interface.encodeFunctionData(
    "transfer",
    [recipientAddress, ethers.parseEther("1000")]
  );

  // 2. 제안 생성
  const proposalTx = await governor.propose(
    [targetContract.address],          // targets
    [0],                               // values (ETH)
    [transferCalldata],                // calldatas
    "Treasury에서 개발팀에 1000 토큰 지급 - QIP-42" // description
  );

  const receipt = await proposalTx.wait();
  const proposalId = receipt.logs[0].args.proposalId;
  console.log(`Proposal ID: ${proposalId}`);

  // 3. 투표 딜레이 대기 후 투표
  // 0 = Against, 1 = For, 2 = Abstain
  await governor.connect(signer).castVoteWithReason(
    proposalId,
    1, // For
    "프로토콜 발전에 기여하는 팀을 지원해야 합니다."
  );

  // 4. 투표 기간 종료 후 큐 등록
  await governor.queue(
    [targetContract.address],
    [0],
    [transferCalldata],
    ethers.id("Treasury에서 개발팀에 1000 토큰 지급 - QIP-42")
  );

  // 5. Timelock 기간 후 실행
  await governor.execute(
    [targetContract.address],
    [0],
    [transferCalldata],
    ethers.id("Treasury에서 개발팀에 1000 토큰 지급 - QIP-42")
  );
}
```

## 주의사항 및 트레이드오프

### 1. 플래시론 공격 (Flash Loan Attack)

가장 위험한 거버넌스 공격 중 하나다. 공격자가 플래시론으로 대량의 토큰을 빌려 단일 트랜잭션 내에서 제안을 통과시키거나 투표를 조작할 수 있다.

**대응책**: ERC20Votes의 스냅샷 메커니즘은 `block.number - 1` 시점의 잔액을 기준으로 하므로, 투표 딜레이를 반드시 1 이상으로 설정해야 한다. `GovernorSettings`에서 `votingDelay`를 0으로 설정하는 것은 절대 금지다.

### 2. 낮은 투표 참여율 (Voter Apathy)

실제 대부분의 DAO에서 투표 참여율은 5~15% 수준이다. Quorum을 너무 높게 설정하면 제안이 영원히 통과되지 않는다.

**트레이드오프**: Quorum을 낮추면 소수 고래가 결정을 좌우한다. 위임(Delegation) 모델을 통해 참여율을 높이되, 위임된 권력의 집중도를 모니터링해야 한다.

### 3. Timelock 기간 설정

| Timelock | 장점 | 단점 |
|----------|------|------|
| 너무 짧음 (< 24h) | 빠른 대응 | 보안 취약, 취소 여유 없음 |
| 적정 (2~7일) | 균형적 | 긴급 상황 대응 어려움 |
| 너무 긺 (> 14일) | 높은 보안 | 운영 민첩성 저하 |

긴급 상황을 위해 **Guardian** 역할을 별도로 두되, 해당 권한이 남용되지 않도록 멀티시그(3-of-5 이상)로 관리해야 한다.

### 4. 업그레이드 가능성과 불변성의 충돌

거버넌스 컨트랙트 자체를 업그레이드할 수 있어야 할까? 업그레이드 가능성은 버그 수정 측면에서 유용하지만, 거버넌스 로직이 변경될 수 있다는 것은 탈중앙화 원칙과 충돌한다.

**권장 패턴**: 거버넌스 컨트랙트는 불변(Immutable)으로 유지하고, 대신 새 거버넌스 컨트랙트를 배포 후 Timelock의 권한을 이전하는 방식으로 "마이그레이션"을 진행한다.

### 5. 거버넌스 공격 시나리오

```
공격 시나리오: 악의적 제안
1. 공격자가 충분한 토큰 확보 (또는 플래시론)
2. 악의적 calldata를 포함한 제안 생성
3. 토큰 홀더들이 제안 내용을 제대로 검토하지 않고 투표
4. Timelock 기간 내 커뮤니티가 취소하지 못함
5. 실행 → 프로토콜 자금 탈취
```

**방어**: 제안 설명과 실제 calldata의 불일치를 자동 감지하는 **온체인 제안 검증기** 또는 Tenderly 같은 시뮬레이션 도구로 제안 실행 결과를 사전 검증하는 프로세스가 필요하다.

## 정리

DAO 거버넌스 설계는 단순히 투표 컨트랙트를 배포하는 것이 아니라, **보안, 참여율, 탈중앙화, 실용성** 사이의 균형을 맞추는 복잡한 시스템 설계 문제다.

핵심 체크리스트를 정리하면 다음과 같다.

- ✅ **Voting Delay > 0**: 플래시론 공격 방어
- ✅ **Timelock 2일 이상**: 악의적 제안 취소 여유 확보
- ✅ **적절한 Quorum**: 너무 높으면 거버넌스 마비
- ✅ **Delegation 지원**: 토큰 홀더 참여율 개선
- ✅ **긴급 Guardian 멀티시그**: 취약점 발견 시 빠른 대응
- ✅ **제안 시뮬레이션 프로세스**: 커뮤니티 리뷰 문화 정착
- ✅ **거버넌스 토큰 분산**: 초기 고래 집중 방지

OpenZeppelin Governor 프레임워크는 이러한 설계 원칙들이 잘 반영되어 있으며, 감사(Audit)를 받은 검증된 코드를 기반으로 하므로 실무에서 처음부터 직접 구현하는 것보다 프레임워크를 확장하는 방식을 강력히 권장한다.

DAO 거버넌스는 코드만큼이나 **커뮤니티 운영 프로세스와 문화**가 중요하다. 아무리 완벽한 스마트 컨트랙트도 참여하지 않는 커뮤니티 앞에서는 무력하다는 점을 항상 염두에 두자.
