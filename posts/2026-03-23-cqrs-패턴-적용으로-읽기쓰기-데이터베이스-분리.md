# CQRS 패턴 적용으로 읽기/쓰기 데이터베이스 분리

## 개요

대규모 서비스를 운영하다 보면 자연스럽게 맞닥뜨리는 문제가 있다. 읽기(Read) 트래픽과 쓰기(Write) 트래픽의 특성이 완전히 다르다는 것이다. 일반적인 서비스에서는 읽기 요청이 쓰기 요청보다 훨씬 많고, 읽기는 복잡한 조인과 집계 쿼리를 필요로 하지만 쓰기는 데이터 무결성과 트랜잭션 일관성을 최우선으로 해야 한다.

단일 데이터베이스로 이 두 가지를 동시에 최적화하는 것은 사실상 불가능에 가깝다. 인덱스를 읽기에 최적화하면 쓰기 성능이 저하되고, 정규화를 강화하면 읽기 쿼리가 복잡해진다.

**CQRS(Command Query Responsibility Segregation)** 패턴은 이 문제에 대한 명쾌한 해답을 제시한다. 명령(Command, 쓰기)과 조회(Query, 읽기)의 책임을 분리함으로써 각각의 목적에 맞게 독립적으로 최적화할 수 있다.

이 글에서는 CQRS의 핵심 개념부터 Spring Boot 환경에서의 실전 구현, 그리고 실무에서 반드시 고려해야 할 트레이드오프까지 깊이 있게 다룬다.

## 핵심 개념

### CQRS란 무엇인가

CQRS는 Greg Young이 CQS(Command Query Separation) 원칙을 아키텍처 레벨로 확장한 패턴이다. 핵심 아이디어는 단순하다.

- **Command**: 시스템의 상태를 변경한다. 반환값이 없거나 단순한 성공/실패 응답만 반환한다.
- **Query**: 시스템의 상태를 조회한다. 사이드 이펙트가 없다.

이 두 책임을 서로 다른 모델, 나아가 서로 다른 데이터베이스로 분리하는 것이 CQRS의 본질이다.

```
┌─────────────────────────────────────────────────────┐
│                    API Gateway                       │
└───────────────────┬─────────────────────────────────┘
                    │
        ┌───────────┴───────────┐
        │                       │
   Command Side            Query Side
        │                       │
┌───────▼───────┐       ┌───────▼───────┐
│  Write Model  │       │  Read Model   │
│   (정규화)     │       │  (비정규화)    │
└───────┬───────┘       └───────┬───────┘
        │                       │
┌───────▼───────┐       ┌───────▼───────┐
│   Write DB    │──────▶│   Read DB     │
│  (PostgreSQL) │  동기화 │  (MongoDB/   │
└───────────────┘       │   Elasticsearch│
                        └───────────────┘
```

### Event Sourcing과의 관계

CQRS는 Event Sourcing과 함께 언급되는 경우가 많지만, 두 패턴은 독립적이다. CQRS 없이 Event Sourcing을 적용할 수 있고, Event Sourcing 없이 CQRS를 적용할 수도 있다. 다만 두 패턴을 함께 사용하면 시너지가 극대화된다.

이 글에서는 Event Sourcing 없이 CQRS를 적용하는 현실적인 방법에 집중한다.

## 실전 예제

### 프로젝트 구성

주문 서비스를 예제로 사용한다. 주문 생성/수정은 Write DB(PostgreSQL), 주문 목록 조회는 Read DB(MongoDB)를 사용하는 구조다.

**의존성 설정 (build.gradle)**

```gradle
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-data-jpa'
    implementation 'org.springframework.boot:spring-boot-starter-data-mongodb'
    implementation 'org.springframework.boot:spring-boot-starter-web'
    implementation 'org.springframework.kafka:spring-kafka'
    runtimeOnly 'org.postgresql:postgresql'
    compileOnly 'org.projectlombok:lombok'
}
```

### DataSource 설정 분리

```java
@Configuration
public class DataSourceConfig {

    @Primary
    @Bean(name = "writeDataSource")
    @ConfigurationProperties(prefix = "spring.datasource.write")
    public DataSource writeDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean(name = "readDataSource")
    @ConfigurationProperties(prefix = "spring.datasource.read")
    public DataSource readDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Primary
    @Bean(name = "writeEntityManagerFactory")
    public LocalContainerEntityManagerFactoryBean writeEntityManagerFactory(
            @Qualifier("writeDataSource") DataSource dataSource,
            JpaProperties jpaProperties) {
        LocalContainerEntityManagerFactoryBean factory = new LocalContainerEntityManagerFactoryBean();
        factory.setDataSource(dataSource);
        factory.setPackagesToScan("com.example.order.domain.write");
        factory.setJpaPropertyMap(jpaProperties.getProperties());
        HibernateJpaVendorAdapter vendorAdapter = new HibernateJpaVendorAdapter();
        factory.setJpaVendorAdapter(vendorAdapter);
        return factory;
    }
}
```

### Command 모델 구현

**도메인 엔티티 (Write Model)**

```java
@Entity
@Table(name = "orders")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String userId;

    @Enumerated(EnumType.STRING)
    private OrderStatus status;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();

    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;

    public static Order create(String userId, List<OrderItemRequest> itemRequests) {
        Order order = new Order();
        order.userId = userId;
        order.status = OrderStatus.PENDING;
        order.createdAt = LocalDateTime.now();
        order.updatedAt = LocalDateTime.now();
        itemRequests.forEach(req -> order.items.add(OrderItem.of(order, req)));
        return order;
    }

    public void confirm() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException("PENDING 상태의 주문만 확정할 수 있습니다.");
        }
        this.status = OrderStatus.CONFIRMED;
        this.updatedAt = LocalDateTime.now();
    }
}
```

**Command 객체와 핸들러**

```java
// Command 객체
@Value
public class CreateOrderCommand {
    String userId;
    List<OrderItemRequest> items;
}

// Command Handler
@Service
@RequiredArgsConstructor
@Transactional
public class OrderCommandService {

    private final OrderRepository orderRepository;
    private final OrderEventPublisher eventPublisher;

    public Long createOrder(CreateOrderCommand command) {
        Order order = Order.create(command.getUserId(), command.getItems());
        Order savedOrder = orderRepository.save(order);

        // Write 완료 후 이벤트 발행 → Read DB 동기화 트리거
        eventPublisher.publish(new OrderCreatedEvent(savedOrder));

        return savedOrder.getId();
    }

    public void confirmOrder(Long orderId) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));
        order.confirm();

        eventPublisher.publish(new OrderConfirmedEvent(order));
    }
}
```

**Query 서비스**

```java
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class OrderQueryService {

    private final OrderViewRepository orderViewRepository;

    public Page<OrderView> getOrdersByUser(String userId, Pageable pageable) {
        return orderViewRepository.findByUserId(userId, pageable);
    }

    public OrderView getOrderDetail(Long orderId) {
        return orderViewRepository.findByOrderId(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));
    }

    public List<OrderView> searchOrders(OrderSearchCondition condition) {
        // MongoDB의 유연한 쿼리 활용
        return orderViewRepository.findByCondition(condition);
    }
}
```

### 이벤트 기반 Read DB 동기화

```java
// Kafka를 통한 이벤트 발행
@Component
@RequiredArgsConstructor
public class OrderEventPublisher {

    private final KafkaTemplate<String, Object> kafkaTemplate;

    public void publish(OrderCreatedEvent event) {
        kafkaTemplate.send("order.created", String.valueOf(event.getOrderId()), event);
    }
}

// Read DB 동기화 Consumer
@Component
@RequiredArgsConstructor
@Slf4j
public class OrderEventConsumer {

    private final OrderViewRepository orderViewRepository;
    private final UserClient userClient;
    private final ProductClient productClient;

    @KafkaListener(topics = "order.created", groupId = "order-read-sync")
    public void handleOrderCreated(OrderCreatedEvent event) {
        try {
            // 여러 서비스에서 데이터를 모아 비정규화된 View 생성
            UserInfo userInfo = userClient.getUserInfo(event.getUserId());
            List<ProductInfo> productInfos = productClient.getProductInfos(
                    event.getItems().stream()
                            .map(OrderItemDto::getProductId)
                            .collect(Collectors.toList())
            );

            OrderView orderView = buildOrderView(event, userInfo, productInfos);
            orderViewRepository.save(orderView);

            log.info("OrderView 동기화 완료: orderId={}", event.getOrderId());
        } catch (Exception e) {
            log.error("OrderView 동기화 실패: orderId={}", event.getOrderId(), e);
            // 재처리 로직 또는 DLQ(Dead Letter Queue) 처리
            throw e;
        }
    }

    private OrderView buildOrderView(OrderCreatedEvent event,
                                     UserInfo userInfo,
                                     List<ProductInfo> productInfos) {
        Map<String, ProductInfo> productMap = productInfos.stream()
                .collect(Collectors.toMap(ProductInfo::getId, Function.identity()));

        List<OrderView.OrderItemView> itemViews = event.getItems().stream()
                .map(item -> OrderView.OrderItemView.builder()
                        .productId(item.getProductId())
                        .productName(productMap.get(item.getProductId()).getName())
                        .quantity(item.getQuantity())
                        .price(item.getPrice())
                        .build())
                .collect(Collectors.toList());

        return OrderView.builder()
                .orderId(event.getOrderId())
                .userId(event.getUserId())
                .userName(userInfo.getName())
                .userEmail(userInfo.getEmail())
                .status(event.getStatus())
                .items(itemViews)
                .totalAmount(calculateTotal(itemViews))
                .createdAt(event.getCreatedAt())
                .updatedAt(event.getCreatedAt())
                .build();
    }
}
```

### 컨트롤러 레이어

```java
@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderCommandService commandService;
    private final OrderQueryService queryService;

    // Command
    @PostMapping
    public ResponseEntity<Long> createOrder(@RequestBody CreateOrderRequest request,
                                             @AuthenticationPrincipal UserDetails user) {
        Long orderId = commandService.createOrder(
                new CreateOrderCommand(user.getUsername(), request.getItems())
        );
        return ResponseEntity.status(HttpStatus.CREATED).body(orderId);
    }

    @PatchMapping("/{orderId}/confirm")
    public ResponseEntity<Void> confirmOrder(@PathVariable Long orderId) {
        commandService.confirmOrder(orderId);
        return ResponseEntity.ok().build();
    }

    // Query
    @GetMapping
    public ResponseEntity<Page<OrderView>> getMyOrders(
            @AuthenticationPrincipal UserDetails user,
            Pageable pageable) {
        return ResponseEntity.ok(queryService.getOrdersByUser(user.getUsername(), pageable));
    }

    @GetMapping("/{orderId}")
    public ResponseEntity<OrderView> getOrderDetail(@PathVariable Long orderId) {
        return ResponseEntity.ok(queryService.getOrderDetail(orderId));
    }
}
```

## 주의사항 및 트레이드오프

### 1. 최종 일관성(Eventual Consistency) 문제

CQRS에서 가장 큰 도전 과제다. Write DB에 저장 후 Read DB에 동기화되기까지 지연이 발생한다. 주문을 생성하고 바로 목록을 조회하면 방금 만든 주문이 보이지 않을 수 있다.

**대응 방안:**
- 클라이언트에게 최종 일관성 모델임을 명확히 안내한다.
- 쓰기 후 응답에 생성된 리소스 ID를 포함해 클라이언트가 폴링할 수 있게 한다.
- 중요한 데이터는 Write DB에서 직접 조회하는 혼합 전략을 사용한다.

### 2. 동기화 실패와 데이터 정합성

네트워크 장애, Consumer 오류 등으로 Read DB 동기화가 실패할 수 있다.

**대응 방안:**
- Kafka DLQ(Dead Letter Queue)를 활용해 실패 이벤트를 보존한다.
- 멱등성(Idempotency)을 보장하는 Consumer를 구현한다.
- 주기적인 배치 작업으로 Write/Read DB 데이터를 비교하고 재동기화한다.

```java
@KafkaListener(topics = "order.created.DLQ")
public void handleFailedEvent(OrderCreatedEvent event) {
    // 알림 발송, 수동 처리 큐 등록 등
    alertService.notify("Read DB 동기화 실패: " + event.getOrderId());
}
```

### 3. 복잡성 증가

단순한 CRUD 애플리케이션에 CQRS를 적용하면 코드량과 운영 복잡도가 크게 증가한다. 다음 기준으로 도입 여부를 결정하자.

| 적합한 경우 | 부적합한 경우 |
|---|---|
| 읽기/쓰기 부하 비율이 극단적으로 다를 때 | 단순 CRUD 서비스 |
| 읽기 모델의 형태가 다양하고 복잡할 때 | 팀의 학습 비용을 감당하기 어려울 때 |
| 마이크로서비스 아키텍처를 사용할 때 | 강한 일관성이 반드시 필요할 때 |
| 독립적인 확장이 필요할 때 | 초기 스타트업 MVP 단계 |

### 4. 운영 복잡도

두 개 이상의 데이터베이스를 운영해야 하므로 모니터링, 장애 대응, 백업 전략 등 운영 부담이 늘어난다. 동기화 지연 지표(Lag Monitoring)를 반드시 구축해야 한다.

## 정리

CQRS는 은탄환(Silver Bullet)이 아니다. 하지만 대규모 트래픽을 처리하는 서비스에서 읽기/쓰기 성능을 각각 독립적으로 최적화해야 할 때, 강력한 해결책이 된다.

핵심을 다시 정리하면:

- **Command(쓰기)**: 정규화된 관계형 DB, 트랜잭션 무결성 보장
- **Query(읽기)**: 비정규화된 NoSQL/검색 엔진, 조회에 최적화된 구조
- **동기화**: 이벤트 기반(Kafka 등) 비동기 처리로 결합도를 낮춤
- **트레이드오프**: 최종 일관성, 운영 복잡도 증가를 반드시 감수해야 함

실무 도입 시에는 전체 시스템에 한 번에 적용하기보다, 병목이 명확한 특정 도메인부터 점진적으로 적용하는 전략을 권장한다. 읽기 성능이 중요한 주문 조회, 상품 검색 등의 기능부터 시작해 팀이 패턴에 익숙해지면 점차 확장하는 것이 안전하다.

CQRS를 제대로 이해하고 적용한다면, 시스템의 확장성과 유지보수성을 동시에 크게 향상시킬 수 있다.
