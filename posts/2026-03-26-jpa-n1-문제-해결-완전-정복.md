# JPA N+1 문제 해결 완전 정복

## 개요

JPA를 실무에서 사용하다 보면 반드시 마주치게 되는 골칫덩어리가 있다. 바로 **N+1 문제**다. 쿼리 하나로 N개의 엔티티를 조회한 후, 연관 엔티티를 가져오기 위해 추가로 N번의 쿼리가 발생하는 현상이다. 로컬 환경에서는 데이터가 적어 눈에 띄지 않다가, 프로덕션에서 데이터가 쌓이면 갑자기 수백 개의 쿼리가 폭발적으로 발생해 성능 장애를 일으킨다.

이 글에서는 N+1 문제가 왜 발생하는지 근본 원인을 짚고, Fetch Join, `@EntityGraph`, Batch Size 등 실무에서 검증된 해결책을 코드와 함께 정리한다. 단순히 "Fetch Join 쓰면 됩니다"에서 끝나지 않고, 각 방법의 트레이드오프까지 다룬다.

---

## 핵심 개념

### N+1 문제가 발생하는 이유

JPA의 연관관계는 기본적으로 **Lazy Loading**으로 동작한다(권장 설정). Lazy Loading은 연관 엔티티를 실제로 사용하는 시점에 프록시를 통해 별도 쿼리를 실행한다.

```java
@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    private List<Member> members = new ArrayList<>();
}

@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    private String username;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "team_id")
    private Team team;
}
```

아래 코드가 실행될 때 문제가 터진다.

```java
List<Team> teams = teamRepository.findAll(); // 쿼리 1번 (팀 N개 조회)

for (Team team : teams) {
    // 각 팀의 members에 접근하는 순간 추가 쿼리 발생
    System.out.println(team.getMembers().size()); // 쿼리 N번
}
// 총 쿼리: 1 + N번
```

`findAll()`로 팀 10개를 가져왔다면 멤버를 조회하기 위해 추가로 10번의 쿼리가 실행된다. 팀이 1000개라면 1001번의 쿼리가 날아간다.

## 실전 예제

### 해결책 1: Fetch Join

가장 직접적인 해결책이다. JPQL에서 `JOIN FETCH`를 사용해 연관 엔티티를 **한 번의 쿼리**로 함께 조회한다.

```java
// Repository
public interface TeamRepository extends JpaRepository<Team, Long> {

    @Query("SELECT t FROM Team t JOIN FETCH t.members")
    List<Team> findAllWithMembers();
}
```

실행되는 SQL:

```sql
SELECT t.*, m.*
FROM team t
INNER JOIN member m ON m.team_id = t.id
```

단 한 번의 쿼리로 팀과 멤버를 모두 가져온다. 명확하고 강력한 방법이지만 주의할 점이 있다(뒤에서 다룬다).

### 해결책 2: @EntityGraph

Spring Data JPA가 제공하는 어노테이션 방식이다. JPQL을 작성하지 않아도 Fetch Join과 동일한 효과를 낸다.

```java
public interface TeamRepository extends JpaRepository<Team, Long> {

    // attributePaths에 즉시 로딩할 연관관계를 명시
    @EntityGraph(attributePaths = {"members"})
    List<Team> findAll();

    // 특정 조건과 함께 사용
    @EntityGraph(attributePaths = {"members"})
    List<Team> findByName(String name);
}
```

엔티티 클래스에 `@NamedEntityGraph`를 선언해 재사용하는 방법도 있다.

```java
@Entity
@NamedEntityGraph(
    name = "Team.withMembers",
    attributeNodes = @NamedAttributeNode("members")
)
public class Team {
    // ...
}

// Repository에서 참조
@EntityGraph("Team.withMembers")
List<Team> findAll();
```

### 해결책 3: Batch Size (BatchFetch)

컬렉션 조회에 특화된 방법이다. N+1 문제를 완전히 없애지는 않지만, N번의 쿼리를 `IN` 절을 활용해 **소수의 쿼리**로 압축한다.

**글로벌 설정 (application.yml):**

```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

**개별 설정 (@BatchSize 어노테이션):**

```java
@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;

    @BatchSize(size = 100)
    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    private List<Member> members = new ArrayList<>();
}
```

팀 100개를 조회하면 기존에는 100번 추가 쿼리가 발생했지만, `default_batch_fetch_size: 100` 설정 시 단 1번의 `IN` 쿼리로 처리된다.

```sql
-- Batch Size 적용 후
SELECT * FROM member WHERE team_id IN (1, 2, 3, ..., 100)
```

### 해결책 4: DTO 직접 조회 (QueryProjection)

연관 엔티티 전체를 엔티티로 가져올 필요가 없을 때 가장 성능이 좋은 방법이다. JPQL이나 QueryDSL로 필요한 데이터만 DTO에 매핑한다.

**DTO 정의:**

```java
@QueryProjection
public record TeamMemberDto(
    Long teamId,
    String teamName,
    String memberUsername
) {}
```

**QueryDSL 적용:**

```java
@Repository
@RequiredArgsConstructor
public class TeamQueryRepository {

    private final JPAQueryFactory queryFactory;

    public List<TeamMemberDto> findTeamWithMembers() {
        return queryFactory
            .select(new QTeamMemberDto(
                team.id,
                team.name,
                member.username
            ))
            .from(team)
            .join(team.members, member)
            .fetch();
    }
}
```

엔티티를 영속성 컨텍스트에 올리지 않고 바로 DTO로 변환하므로, 불필요한 데이터 로딩이 없고 성능이 가장 뛰어나다.

## 주의사항 및 트레이드오프

### Fetch Join의 한계: 컬렉션 페이징 불가

`@OneToMany` 컬렉션에 Fetch Join을 적용하면서 **페이징**을 함께 사용하면 Hibernate가 경고를 내뿜으며 **모든 데이터를 메모리에 올린 후 페이징**한다.

```
HHH90003004: firstResult/maxResults specified with collection fetch;
applying in memory
```

이는 데이터가 많을 경우 **OOM(Out Of Memory)**으로 이어질 수 있는 심각한 문제다.

**컬렉션 페이징이 필요할 때의 해결책:**

```java
// ToOne 관계는 Fetch Join으로 가져오고,
// 컬렉션은 Batch Size로 처리하는 전략
@Query("SELECT t FROM Team t JOIN FETCH t.leader WHERE t.active = true")
Page<Team> findActiveTeams(Pageable pageable);
```

`@ManyToOne`, `@OneToOne` 같은 **ToOne 관계**는 Fetch Join + 페이징을 함께 써도 안전하다. **컬렉션(OneToMany, ManyToMany)에서만 문제가 발생**한다는 점을 명심하자.

### Fetch Join 중복 데이터 문제

`OneToMany` Fetch Join은 조인 특성상 팀 1개에 멤버가 5명이면 팀 엔티티가 5번 중복 조회된다.

```java
// DISTINCT 키워드로 중복 제거
@Query("SELECT DISTINCT t FROM Team t JOIN FETCH t.members")
List<Team> findAllWithMembers();
```

Hibernate 6부터는 `HibernateJpaDialect`가 자동으로 중복을 제거하지만, 명시적으로 `DISTINCT`를 사용하는 습관을 들이는 것이 좋다.

### 다중 컬렉션 Fetch Join 금지

두 개 이상의 컬렉션에 동시에 Fetch Join을 적용하면 **MultipleBagFetchException**이 발생한다.

```java
// 이렇게 하면 안 된다!
@Query("SELECT t FROM Team t JOIN FETCH t.members JOIN FETCH t.projects")
List<Team> findAllWithMembersAndProjects(); // 예외 발생!
```

이 경우에는 하나의 컬렉션만 Fetch Join하고 나머지는 **Batch Size**로 처리하거나, `List` 대신 `Set`을 사용하는 방법을 고려한다.

```java
// 권장 패턴: 하나는 Fetch Join, 나머지는 Batch Size
@Entity
public class Team {
    @BatchSize(size = 100)
    @OneToMany(mappedBy = "team")
    private List<Member> members; // Batch Size로 처리

    @OneToMany(mappedBy = "team")
    private List<Project> projects; // 별도 Fetch Join
}
```

### @EntityGraph vs Fetch Join 선택 기준

| 구분 | Fetch Join | @EntityGraph |
|------|-----------|-------------|
| 코드 복잡도 | JPQL 작성 필요 | 어노테이션만으로 간결 |
| 유연성 | 복잡한 조건 가능 | 단순 조회에 적합 |
| 동적 쿼리 | QueryDSL 조합 필요 | 동적 조합 어려움 |
| 내부 동작 | 동일 (LEFT OUTER JOIN) | 동일 (LEFT OUTER JOIN) |

단순한 조회 메서드에는 `@EntityGraph`가 깔끔하고, 복잡한 조건이 붙거나 QueryDSL을 사용한다면 Fetch Join을 선택하자.

### 해결책별 적합한 사용 상황 요약

```
조회 결과를 엔티티로 반환 + 페이징 없음
  → Fetch Join 또는 @EntityGraph

조회 결과를 엔티티로 반환 + 페이징 있음
  → ToOne은 Fetch Join, 컬렉션은 Batch Size

읽기 전용 API, 성능이 매우 중요
  → DTO 직접 조회 (QueryDSL Projection)

다중 컬렉션 조회
  → 하나만 Fetch Join + 나머지는 Batch Size
```

## 정리

N+1 문제는 JPA를 올바르게 이해하지 않으면 언제든 다시 발생할 수 있는 고질적인 문제다. 핵심을 정리하면 다음과 같다.

1. **Fetch Join**은 가장 직관적인 해결책이지만, 컬렉션 페이징과 함께 쓰면 안 된다.
2. **@EntityGraph**는 JPQL 없이 Fetch Join과 동일한 효과를 내며, 단순 조회에 적합하다.
3. **Batch Size**는 컬렉션 페이징 시 필수 전략이며, `default_batch_fetch_size`를 100~1000 사이로 글로벌 설정해두면 안전망 역할을 한다.
4. **DTO 직접 조회**는 성능이 가장 뛰어나며, 읽기 전용 API에 적극 활용해야 한다.

실무에서는 단일 전략을 고집하지 말고, 상황에 따라 조합해서 사용하는 것이 핵심이다. 개발 단계에서 **Hibernate의 SQL 로그를 반드시 활성화**해 실제로 몇 번의 쿼리가 발생하는지 눈으로 확인하는 습관을 들이자. 쿼리 로그를 보지 않으면 N+1 문제는 조용히 프로덕션까지 숨어든다.

```yaml
# 개발 환경 SQL 로그 설정
logging:
  level:
    org.hibernate.SQL: debug
    org.hibernate.orm.jdbc.bind: trace
```

N+1 문제를 정복하는 것은 JPA를 제대로 다루는 개발자의 기본기다. 이 글에서 다룬 전략들을 실무에 맞게 적용해 불필요한 쿼리를 줄이고 안정적인 서비스를 만들어 나가길 바란다.
