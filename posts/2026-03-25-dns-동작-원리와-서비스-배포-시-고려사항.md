# DNS 동작 원리와 서비스 배포 시 고려사항

## 개요

서비스를 배포하고 나서 "왜 내 도메인이 아직도 안 열리지?" 혹은 "A/B 환경 전환을 했는데 일부 사용자만 새 서버로 붙는다"는 경험을 해본 적이 있을 것이다. 이런 현상은 대부분 DNS(Domain Name System)의 동작 방식을 정확히 이해하지 못한 데서 비롯된다.

DNS는 인터넷의 전화번호부로 자주 비유되지만, 실무에서는 **캐싱 계층, TTL, 전파 지연, 레코드 타입** 등 훨씬 복잡한 개념들이 얽혀 있다. 특히 배포 자동화, 무중단 배포, 멀티 리전 아키텍처를 다루는 백엔드 엔지니어라면 DNS의 내부 동작을 정확히 이해하고 있어야 장애를 예방하고 빠르게 대응할 수 있다.

이 글에서는 DNS의 핵심 동작 원리부터 실제 서비스 배포 시 마주치는 문제들과 해결책을 다룬다.

## 핵심 개념

### DNS 조회 흐름 (Recursive Resolution)

DNS 조회는 단순히 "도메인 → IP"가 아니라 여러 서버를 거치는 계층적 프로세스다.

```
클라이언트
  └─▶ Local DNS Cache (OS)
        └─▶ Recursive Resolver (ISP 또는 8.8.8.8)
              └─▶ Root Name Server (.)
                    └─▶ TLD Name Server (.com, .io 등)
                          └─▶ Authoritative Name Server (실제 레코드 보유)
```

1. **클라이언트**: OS의 로컬 캐시 확인 → `/etc/hosts` 확인
2. **Recursive Resolver**: ISP가 제공하거나 `8.8.8.8`, `1.1.1.1` 같은 공개 리졸버. 캐시가 없으면 루트까지 올라가 위임 체인을 따라 내려온다.
3. **Authoritative Name Server**: 최종적으로 실제 DNS 레코드를 반환하는 서버. Route 53, Cloudflare DNS, GCP Cloud DNS 등이 여기에 해당한다.

### 주요 DNS 레코드 타입

| 레코드 | 용도 | 예시 |
|--------|------|------|
| A | 도메인 → IPv4 | `api.example.com → 1.2.3.4` |
| AAAA | 도메인 → IPv6 | `api.example.com → ::1` |
| CNAME | 도메인 → 다른 도메인 | `www → example.com` |
| MX | 메일 서버 지정 | 이메일 라우팅 |
| TXT | 텍스트 데이터 | SSL 인증, SPF/DKIM |
| NS | Authoritative NS 지정 | 도메인 위임 |
| SRV | 서비스 위치 | gRPC, SIP 등 |
| ALIAS/ANAME | CNAME의 루트 도메인 버전 | Route 53 Alias |

### TTL(Time To Live)의 이중성

TTL은 "캐시 유효 시간(초)"이다. TTL이 낮을수록 변경 사항이 빠르게 전파되지만 DNS 쿼리 수가 늘어나고, 높을수록 캐시 효율이 좋지만 변경 반영이 느리다.

```bash
# 현재 TTL 확인
dig api.example.com +noall +answer

# 결과 예시
# api.example.com.  300  IN  A  1.2.3.4
#                   ^^^
#                   TTL = 300초 (5분)
```

**중요한 오해**: TTL이 0이어도 Recursive Resolver가 최소 캐시 시간을 강제하는 경우가 있다. Cloudflare는 최소 60초, 일부 ISP 리졸버는 30~60초를 강제 적용한다.

## 실전 예제

### 예제 1: 배포 전 TTL 선제 조정

무중단 배포 전날, TTL을 낮춰두는 것이 핵심 전략이다. 기존 TTL이 86400(24시간)이면 변경 후에도 최대 24시간 동안 구 캐시가 유효하다.

```bash
# 현재 TTL 확인 (배포 전)
dig +short api.example.com

# Route 53 CLI로 TTL 변경 (300초 → 60초)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "api.example.com",
        "Type": "A",
        "TTL": 60,
        "ResourceRecords": [{"Value": "1.2.3.4"}]
      }
    }]
  }'
```

**권장 절차**:
1. 배포 T-24h: TTL을 현재 값의 1/10로 낮춤 (86400 → 300)
2. 배포 T-1h: TTL을 60초로 낮춤
3. 배포 수행 + IP 전환
4. 안정화 확인 후: TTL을 다시 높임 (300 → 3600)

### 예제 2: Java에서 DNS 캐시 제어

JVM은 자체적인 DNS 캐시를 보유한다. 이를 모르면 서버 IP가 바뀌어도 애플리케이션이 구 IP로 계속 연결할 수 있다.

```java
// JVM DNS 캐시 TTL 확인 및 설정
// 기본값: networkaddress.cache.ttl = -1 (무한 캐시, 보안상 이유)
//         networkaddress.cache.negative.ttl = 10

import java.security.Security;

public class DnsCacheConfig {
    
    static {
        // 양성 결과 캐시: 60초
        Security.setProperty("networkaddress.cache.ttl", "60");
        // 음성 결과 캐시 (NXDOMAIN 등): 5초
        Security.setProperty("networkaddress.cache.negative.ttl", "5");
    }
    
    public static void main(String[] args) throws Exception {
        // 현재 설정 확인
        System.out.println("Positive TTL: " + 
            Security.getProperty("networkaddress.cache.ttl"));
        System.out.println("Negative TTL: " + 
            Security.getProperty("networkaddress.cache.negative.ttl"));
    }
}
```

```yaml
# Spring Boot application.yml 환경에서
# JVM 옵션으로 전달하는 방법 (Dockerfile 또는 실행 스크립트)
# JAVA_OPTS: -Dnetworkaddress.cache.ttl=60 -Dnetworkaddress.cache.negative.ttl=5
```

```properties
# java.security 파일에서 전역 설정 (권장하지 않음 - 보안 고려 필요)
networkaddress.cache.ttl=60
networkaddress.cache.negative.ttl=5
```

> **주의**: `networkaddress.cache.ttl=-1`(기본값)은 JVM 실행 중 DNS 레코드가 바뀌어도 감지 못한다. 클라우드 환경, ECS/Kubernetes에서는 반드시 적절한 값으로 설정해야 한다.

### 예제 3: Blue/Green 배포에서 DNS 활용

```bash
# Blue 환경 (현재 운영)
# api.example.com → blue.example.com (CNAME)
# blue.example.com → 10.0.1.10 (A)

# Green 환경 (신규 배포)
# green.example.com → 10.0.2.10 (A)

# 전환: CNAME 타겟만 변경
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "api.example.com",
        "Type": "CNAME",
        "TTL": 60,
        "ResourceRecords": [{"Value": "green.example.com"}]
      }
    }]
  }'

# 롤백 시: blue.example.com으로 되돌리기만 하면 됨
```

### 예제 4: DNS 기반 헬스체크 (Route 53)

```json
// Route 53 Health Check + Failover 레코드 설정 예시
{
  "HealthCheckConfig": {
    "IPAddress": "10.0.1.10",
    "Port": 8080,
    "Type": "HTTP",
    "ResourcePath": "/actuator/health",
    "RequestInterval": 10,
    "FailureThreshold": 3,
    "EnableSNI": false
  }
}
```

```bash
# Failover 레코드 구성 (Primary / Secondary)
# Primary가 헬스체크 실패 시 Secondary IP로 자동 전환

# Primary Record
aws route53 change-resource-record-sets --change-batch '{
  "Changes": [{
    "Action": "UPSERT",
    "ResourceRecordSet": {
      "Name": "api.example.com",
      "Type": "A",
      "SetIdentifier": "Primary",
      "Failover": "PRIMARY",
      "TTL": 30,
      "ResourceRecords": [{"Value": "10.0.1.10"}],
      "HealthCheckId": "abc-health-check-id"
    }
  }]
}'
```

### 예제 5: DNS 조회 디버깅 도구

```bash
# 기본 조회
dig api.example.com

# 특정 레코드 타입 조회
dig api.example.com MX
dig api.example.com TXT

# 특정 DNS 서버 지정 (ISP 캐시 우회)
dig @8.8.8.8 api.example.com
dig @1.1.1.1 api.example.com

# Authoritative NS에 직접 조회 (가장 최신 정보)
dig api.example.com NS
dig @ns1.example-dns.com api.example.com

# 전체 위임 체인 추적
dig +trace api.example.com

# 응답 시간 측정
dig api.example.com | grep "Query time"

# macOS: dscacheutil로 로컬 캐시 확인/초기화
sudo dscacheutil -flushcache && sudo killall -HUP mDNSResponder

# Linux: systemd-resolve 캐시 초기화
sudo systemd-resolve --flush-caches
```

## 주의사항 및 트레이드오프

### 1. DNS Propagation은 예측 불가능하다

"DNS 전파 시간은 최대 48시간"이라는 말은 반은 맞고 반은 틀리다. 정확히는 **각 Recursive Resolver의 캐시 TTL이 만료되어야** 새 레코드를 가져간다. 

- ISP 리졸버가 TTL을 무시하고 더 오래 캐시하는 경우가 있다.
- Anycast 기반 공개 DNS(1.1.1.1, 8.8.8.8)는 보통 수 분 내 반영된다.
- 오래된 기업망 DNS는 수 시간 이상 캐시할 수 있다.

**대응**: 배포 전 충분히 TTL을 낮추는 것 외에 완벽한 해결책은 없다. Blue/Green + 로드밸런서 레벨 라우팅을 조합하면 DNS 의존도를 줄일 수 있다.

### 2. CNAME은 루트 도메인에 사용할 수 없다

`example.com` (apex 도메인)에는 CNAME을 설정할 수 없다 — RFC 1034 제약이다. AWS ALB나 CloudFront를 루트 도메인에 연결하려면 **Route 53 Alias 레코드** 또는 **ANAME**을 사용해야 한다.

```bash
# 잘못된 설정 (불가)
# example.com CNAME alb-xxx.ap-northeast-2.elb.amazonaws.com

# 올바른 설정 (Route 53 Alias)
# example.com ALIAS alb-xxx.ap-northeast-2.elb.amazonaws.com
# - TTL 없음 (AWS가 내부적으로 관리)
# - 헬스체크 연동 가능
```

### 3. 낮은 TTL = 비용 증가

Route 53은 쿼리 수 기반 과금이다. 트래픽이 많은 서비스에서 TTL을 무조건 낮추면 DNS 쿼리 비용이 급증한다. **평상시 TTL은 300~3600초를 유지**하고 배포 이벤트 전후로만 낮추는 전략이 합리적이다.

### 4. Split-Horizon DNS 복잡성

내부망과 외부망에서 같은 도메인이 다른 IP를 반환해야 하는 경우(Split-Horizon) 설정 관리가 복잡해진다. Kubernetes 환경에서는 CoreDNS의 `rewrite` 플러그인이나 별도 내부 DNS 존을 활용한다.

```yaml
# CoreDNS ConfigMap 예시 (내부 서비스 오버라이드)
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health
        rewrite name api.example.com api-service.default.svc.cluster.local
        kubernetes cluster.local in-addr.arpa ip6.arpa {
            pods insecure
            fallthrough in-addr.arpa ip6.arpa
        }
        forward . /etc/resolv.conf
        cache 30
        loop
        reload
        loadbalance
    }
```

### 5. DNSSEC와 성능 트레이드오프

DNSSEC는 DNS 응답의 무결성을 보장하지만 서명 검증 오버헤드로 응답 지연이 발생할 수 있다. 금융, 공공 서비스는 필수이지만 일반 서비스에서는 CDN의 TLS 기반 보호로 대체하는 경우가 많다.

## 정리

DNS는 단순한 "이름 해석기"가 아니라 서비스 가용성, 배포 전략, 트래픽 라우팅의 핵심 인프라다.

| 상황 | 권장 사항 |
|------|-----------|
| 배포 전 | TTL을 최소 1 TTL 주기 전에 낮춰둘 것 |
| JVM 서비스 | `networkaddress.cache.ttl` 명시적으로 설정 |
| Blue/Green | CNAME + 낮은 TTL 조합으로 빠른 전환 |
| Apex 도메인 | CNAME 대신 Alias/ANAME 사용 |
| 장애 대응 | DNS Failover + 헬스체크 자동화 |
| 내부 라우팅 | CoreDNS rewrite 또는 별도 내부 존 |

DNS를 올바르게 이해하고 설정하면 배포 리스크를 크게 줄이고 장애 복구 시간을 단축할 수 있다. 다음 배포 전에 `dig +trace`로 현재 레코드 상태를 한번 점검해보자.
