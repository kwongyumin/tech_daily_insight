# QUIC 프로토콜과 HTTP/3의 등장 배경과 특징

## 개요

웹이 점점 복잡해지고 실시간성이 중요해지면서 기존 HTTP/1.1과 HTTP/2가 가진 근본적인 한계가 수면 위로 떠오르기 시작했다. 특히 모바일 환경이 보편화되면서 네트워크 전환(Wi-Fi ↔ LTE)이나 패킷 손실 상황에서 기존 프로토콜 스택의 취약점이 극명하게 드러났다.

Google은 이 문제를 해결하기 위해 2012년부터 QUIC(Quick UDP Internet Connections) 프로토콜을 실험적으로 개발하기 시작했고, 2021년 IETF가 이를 표준화하면서 HTTP/3의 기반 전송 계층으로 공식 채택했다. 현재 Google, Meta, Cloudflare 등 대형 서비스들은 이미 HTTP/3를 프로덕션에서 적극 활용하고 있다.

이 포스팅에서는 QUIC과 HTTP/3가 왜 필요했는지 그 배경부터, 핵심 기술 개념, 실무 적용 방법까지 깊이 있게 다뤄보겠다.

---

## 기존 프로토콜의 한계: 왜 QUIC이 필요했는가?

### TCP의 구조적 문제

HTTP/1.1과 HTTP/2는 모두 TCP 위에서 동작한다. TCP는 신뢰성 있는 전송을 보장하지만, 이 신뢰성이 오히려 성능 병목을 만든다.

**HOL Blocking (Head-of-Line Blocking)**이 대표적인 문제다. HTTP/2는 멀티플렉싱을 통해 하나의 TCP 연결에서 여러 스트림을 동시에 처리할 수 있지만, 하나의 TCP 세그먼트가 손실되면 해당 연결의 **모든 스트림이 재전송을 기다리며 블로킹**된다. 이것이 TCP 레벨의 HOL Blocking이다.

```
HTTP/2 with TCP:
Stream 1: [A1][A2][  ][A4]  <- 패킷 손실 발생
Stream 2: [B1][B2][??][B4]  <- B3 도착했지만 대기
Stream 3: [C1][C2][??][C4]  <- C3 도착했지만 대기

→ 하나의 TCP 패킷 손실이 모든 스트림을 블로킹
```

**느린 연결 수립(Handshake Overhead)** 또한 문제다. TCP는 기본적으로 3-way handshake가 필요하고, HTTPS를 위한 TLS 1.2 협상까지 더하면 데이터를 주고받기 전에 **최소 2~3 RTT**가 소요된다.

```
TCP + TLS 1.2:
Client → Server: SYN                     (1 RTT)
Client ← Server: SYN-ACK
Client → Server: ACK

Client → Server: ClientHello             (2 RTT)
Client ← Server: ServerHello + Cert
Client → Server: ClientKeyExchange + Finished
Client ← Server: Finished

→ 총 2 RTT 후 데이터 전송 가능
```

**Connection Migration 불가** 문제도 있다. TCP 연결은 4-tuple(src IP, src Port, dst IP, dst Port)로 식별된다. 모바일 기기가 Wi-Fi에서 LTE로 전환하면 src IP가 바뀌어 기존 연결이 끊기고 재연결이 필요하다.

## QUIC의 핵심 개념

### 1. UDP 기반의 신뢰성 레이어

QUIC은 **UDP를 기반**으로 하지만, TCP처럼 신뢰성 있는 전송을 애플리케이션 레이어에서 직접 구현한다. UDP를 선택한 이유는 단순하다. 기존 네트워크 장비들(NAT, 방화벽, 미들박스)이 TCP의 동작 방식에 깊이 의존하고 있어 TCP를 수정하기가 현실적으로 불가능하기 때문이다.

### 2. 0-RTT / 1-RTT 연결 수립

QUIC은 TLS 1.3을 전송 계층에 통합하여 연결 수립과 암호화 협상을 동시에 처리한다.

```
QUIC + TLS 1.3 (초기 연결):
Client → Server: Initial Packet (CRYPTO, ClientHello) → 1 RTT
Client ← Server: Initial + Handshake Packet (ServerHello, Finished)
Client → Server: Handshake Finished + HTTP Request
Client ← Server: HTTP Response

→ 1 RTT 후 데이터 전송 가능

QUIC (재연결, 0-RTT):
Client → Server: Initial Packet + 0-RTT Data (이전 세션 정보 재사용)
Client ← Server: HTTP Response

→ 0 RTT, 첫 패킷부터 데이터 포함 가능
```

### 3. 스트림 레벨의 독립적 흐름 제어

QUIC의 가장 핵심적인 혁신이다. HTTP/2와 달리 QUIC에서는 **각 스트림이 독립적으로 패킷 손실을 처리**한다. 특정 스트림의 패킷이 손실되어도 다른 스트림은 영향받지 않는다.

```
QUIC Multiplexing:
Stream 1: [A1][A2][  ][A4]  <- 패킷 손실 → Stream 1만 재전송 대기
Stream 2: [B1][B2][B3][B4]  <- 정상 처리
Stream 3: [C1][C2][C3][C4]  <- 정상 처리

→ HOL Blocking 해소
```

### 4. Connection ID 기반 연결 유지 (Connection Migration)

QUIC은 4-tuple 대신 **Connection ID**를 기반으로 연결을 식별한다. 네트워크가 변경되어 IP나 포트가 바뀌더라도 Connection ID가 동일하면 연결을 유지할 수 있다.

## 실전 예제

### Nginx에서 HTTP/3 활성화

최신 Nginx(1.25+)는 HTTP/3를 공식 지원한다. 아래는 기본 설정 예제다.

```nginx
# /etc/nginx/nginx.conf

http {
    server {
        listen 443 ssl;
        listen 443 quic reuseport;  # HTTP/3 (QUIC) 리스닝 추가

        server_name example.com;

        ssl_certificate     /etc/ssl/certs/example.com.crt;
        ssl_certificate_key /etc/ssl/private/example.com.key;

        # TLS 1.3 필수 (QUIC은 TLS 1.3만 지원)
        ssl_protocols TLSv1.3;

        # HTTP/3 지원을 클라이언트에게 알림
        add_header Alt-Svc 'h3=":443"; ma=86400';
        add_header QUIC-Status $quic;

        location / {
            proxy_pass http://backend;
        }
    }
}
```

### Spring Boot에서 HTTP/3 클라이언트 사용 (OkHttp)

Java 백엔드에서 HTTP/3 클라이언트를 사용하려면 현재 가장 성숙한 옵션은 OkHttp 5.x + Conscrypt 조합이다.

```java
// build.gradle
dependencies {
    implementation 'com.squareup.okhttp3:okhttp:5.0.0-alpha.12'
    implementation 'org.conscrypt:conscrypt-openjdk-uber:2.5.2'
}
```

```java
import okhttp3.*;
import org.conscrypt.Conscrypt;
import java.security.Security;

@Configuration
public class Http3ClientConfig {

    @Bean
    public OkHttpClient http3Client() {
        // Conscrypt를 최우선 보안 프로바이더로 등록 (TLS 1.3 QUIC 지원)
        Security.insertProviderAt(Conscrypt.newProvider(), 1);

        return new OkHttpClient.Builder()
                .protocols(List.of(Protocol.QUIC, Protocol.HTTP_2, Protocol.HTTP_1_1))
                .connectTimeout(Duration.ofSeconds(10))
                .readTimeout(Duration.ofSeconds(30))
                .build();
    }
}
```

```java
@Service
@RequiredArgsConstructor
public class Http3ExampleService {

    private final OkHttpClient http3Client;

    public String fetchData(String url) throws IOException {
        Request request = new Request.Builder()
                .url(url)
                .header("User-Agent", "MyApp/1.0")
                .build();

        try (Response response = http3Client.newCall(request).execute()) {
            // 실제 사용된 프로토콜 확인
            System.out.println("Protocol used: " + response.protocol());
            // h3 → HTTP/3, h2 → HTTP/2, http/1.1 → HTTP/1.1

            if (!response.isSuccessful()) {
                throw new IOException("Unexpected response: " + response);
            }

            return response.body().string();
        }
    }
}
```

### cURL로 HTTP/3 테스트

```bash
# HTTP/3 지원 확인 (curl 7.66.0+)
curl --http3 -I https://example.com

# 응답 헤더에서 Alt-Svc 확인
curl -I https://cloudflare.com | grep -i alt-svc
# alt-svc: h3=":443"; ma=86400

# 상세 연결 정보 출력
curl --http3 -v --write-out "\n\nProtocol: %{http_version}\n" https://example.com
```

### Wireshark로 QUIC 패킷 분석

```bash
# tcpdump로 QUIC 트래픽 캡처 (UDP 443)
sudo tcpdump -i eth0 -w quic_capture.pcap 'udp port 443'

# Wireshark 필터
# QUIC 패킷만 보기
quic

# 특정 Connection ID 필터
quic.dcid == "abcd1234..."

# 스트림별 분석
quic.stream.stream_id == 0
```

## 주의사항 및 트레이드오프

### 1. UDP 차단 문제

기업 방화벽이나 일부 ISP에서 UDP 443 포트를 차단하는 경우가 있다. 실제로 전체 인터넷 트래픽의 약 5~10%에서 QUIC이 차단된다는 통계가 있다. 이 때문에 **HTTP/3는 반드시 HTTP/2 fallback과 함께 구성**해야 한다.

```nginx
# Alt-Svc 헤더를 통해 클라이언트가 HTTP/3를 시도하고
# 실패하면 HTTP/2로 자동 폴백
add_header Alt-Svc 'h3=":443"; ma=86400, h2=":443"';
```

### 2. CPU 오버헤드

QUIC은 커널 레벨에서 처리되는 TCP와 달리 **유저스페이스에서 구현**된다. 암호화, 혼잡 제어, 재전송 로직을 모두 애플리케이션이 처리하기 때문에 CPU 사용량이 TCP 대비 높을 수 있다. Google 내부 벤치마크에서는 동일 처리량 기준으로 QUIC이 TCP보다 CPU를 약 2배 더 사용한다는 결과도 있다.

### 3. 0-RTT의 보안 취약점

0-RTT 재연결은 강력한 성능 최적화이지만 **재전송 공격(Replay Attack)에 취약**하다. 0-RTT 데이터를 공격자가 캡처하여 재전송하면 서버가 이를 재처리할 수 있다. 따라서 **0-RTT는 멱등성이 보장되는 GET 요청에만 사용**하고, 상태를 변경하는 POST/PUT/DELETE에는 사용하지 않아야 한다.

```java
// 잘못된 예: 0-RTT로 결제 요청 처리
// → 재전송 공격 시 이중 결제 발생 가능

// 올바른 예: 0-RTT는 캐시 조회 등 읽기 전용 요청에만
Request safeRequest = new Request.Builder()
        .url("https://api.example.com/products/123")
        .get()  // GET만 0-RTT 허용
        .build();
```

### 4. 로드밸런서 호환성

QUIC의 Connection Migration은 IP가 바뀌어도 연결을 유지한다. 하지만 기존 IP 해시 기반 로드밸런서는 클라이언트 IP 변경 시 다른 백엔드로 라우팅할 수 있다. **QUIC 지원 로드밸런서는 Connection ID 기반 스티키 세션을 구현**해야 한다.

### 5. 모니터링의 어려움

QUIC은 페이로드뿐 아니라 **헤더까지 암호화**되어 있어 기존 패킷 분석 도구로는 내용을 볼 수 없다. 개발 환경에서 디버깅이 필요하다면 `SSLKEYLOGFILE` 환경변수를 활용해 TLS 키를 추출하여 Wireshark에서 복호화할 수 있다.

```bash
# 개발 환경에서만 사용 (프로덕션 절대 금지)
export SSLKEYLOGFILE=/tmp/tls_keys.log
curl --http3 https://example.com

# Wireshark에서 Edit > Preferences > Protocols > TLS
# (Pre)-Master-Secret log filename에 위 경로 입력
```

## 정리

QUIC과 HTTP/3는 단순한 버전 업그레이드가 아니다. 수십 년 된 TCP 스택의 근본적인 설계 한계를 해결하기 위한 **전송 계층의 패러다임 전환**이다.

핵심을 정리하면 다음과 같다:

| 항목 | TCP + HTTP/2 | QUIC + HTTP/3 |
|------|-------------|---------------|
| 전송 계층 | TCP | UDP (QUIC) |
| 연결 수립 | 2 RTT (TLS 1.2) | 1 RTT / 0 RTT |
| HOL Blocking | TCP 레벨 존재 | 스트림 독립 처리 |
| Connection Migration | 불가 | Connection ID로 지원 |
| 암호화 | TLS 선택적 | TLS 1.3 내장 필수 |
| 구현 위치 | 커널 | 유저스페이스 |

실무 적용 관점에서는 **엣지/CDN 레이어(Cloudflare, AWS CloudFront)에서 HTTP/3를 먼저 활성화**하고, 오리진 서버는 HTTP/2를 유지하는 하이브리드 전략이 현실적이다. 클라이언트와 CDN 구간에서 QUIC의 이점을 누리면서, 안정성이 검증된 TCP 기반 내부 통신은 유지하는 방식이다.

인터넷 트래픽의 상당 부분이 이미 HTTP/3로 전환되고 있는 지금, 이 프로토콜에 대한 이해는 단순한 선택이 아니라 백엔드 엔지니어의 필수 역량이 되어가고 있다.
