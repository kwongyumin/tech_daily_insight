# 데이터베이스 레플리케이션과 읽기 분산 전략

## 개요

트래픽이 증가하면 데이터베이스는 가장 먼저 병목이 되는 지점 중 하나다. 특히 읽기(READ) 요청이 쓰기(WRITE) 요청보다 압도적으로 많은 일반적인 서비스 특성상, 단일 데이터베이스 인스턴스로는 한계에 부딪히는 순간이 반드시 온다.

**데이터베이스 레플리케이션(Replication)** 은 이 문제를 해결하기 위한 가장 검증된 방법 중 하나다. Primary(Master) 노드에서 모든 쓰기를 처리하고, 하나 이상의 Replica(Slave) 노드가 데이터를 복제해 읽기 요청을 분산 처리하는 구조다.

이 글에서는 레플리케이션의 핵심 개념부터 Spring Boot 환경에서의 실전 구현, 그리고 운영 시 반드시 고려해야 할 트레이드오프까지 실무 관점에서 정리한다.

## 핵심 개념

### 레플리케이션 동작 방식

MySQL 기준으로 레플리케이션은 **바이너리 로그(Binary Log)** 기반으로 동작한다.

1. Primary가 데이터 변경 시 Binary Log에 기록
2. Replica의 **I/O Thread**가 Binary Log를 읽어 **Relay Log**에 저장
3. Replica의 **SQL Thread**가 Relay Log를 읽어 실제 데이터에 반영

```
Primary                          Replica
┌─────────────┐                 ┌─────────────────┐
│  Write/Read │                 │    Read Only    │
│             │  Binary Log     │                 │
│  Bin Log ───┼────────────────►│ Relay Log       │
│             │                 │     │           │
└─────────────┘                 │     ▼           │
                                │ SQL Thread      │
                                │ (Apply Changes) │
                                └─────────────────┘
```

### 동기(Sync) vs 비동기(Async) 복제

| 구분 | 동기 복제 | 비동기 복제 |
|------|-----------|-------------|
| 데이터 일관성 | 강함 | 약함 (Replication Lag 발생) |
| 성능 영향 | 쓰기 지연 증가 | 쓰기 성능 영향 최소 |
| 장애 시 데이터 손실 | 없음 | 발생 가능 |
| 대표 사례 | MySQL Group Replication, Galera | MySQL 기본 비동기 복제 |

실무에서는 **반동기(Semi-synchronous)** 복제도 많이 사용한다. 적어도 하나의 Replica가 수신을 확인하면 Primary가 커밋을 완료하는 방식으로, 성능과 안정성의 균형점을 찾는다.

### Replication Lag (복제 지연)

비동기 복제의 가장 큰 함정이다. Primary에 데이터를 쓴 직후 Replica에서 읽으면 이전 데이터가 조회될 수 있다. 이를 **Read-Your-Writes Consistency** 문제라 부른다.

```
t=0: Primary에 user_id=1 INSERT
t=0.1: Replica에 아직 복제 안 됨 (Lag = 100ms)
t=0.05: Replica에서 user_id=1 조회 → 결과 없음 ← 문제!
```

## 실전 예제 — Spring Boot에서 읽기/쓰기 DB 분기

### 1. 의존성 및 설정

```yaml
# application.yml
spring:
  datasource:
    primary:
      url: jdbc:mysql://primary-db:3306/mydb
      username: root
      password: secret
      driver-class-name: com.mysql.cj.jdbc.Driver
    replica:
      url: jdbc:mysql://replica-db:3306/mydb
      username: readonly
      password: secret
      driver-class-name: com.mysql.cj.jdbc.Driver
```

### 2. DataSource 설정 — AbstractRoutingDataSource 활용

Spring의 `AbstractRoutingDataSource`를 사용하면 트랜잭션의 읽기/쓰기 여부에 따라 DataSource를 동적으로 라우팅할 수 있다.

```java
// DataSourceType.java
public enum DataSourceType {
    PRIMARY, REPLICA
}
```

```java
// RoutingDataSource.java
public class RoutingDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        boolean isReadOnly = TransactionSynchronizationManager.isCurrentTransactionReadOnly();
        return isReadOnly ? DataSourceType.REPLICA : DataSourceType.PRIMARY;
    }
}
```

```java
// DataSourceConfig.java
@Configuration
public class DataSourceConfig {

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource.primary")
    public DataSource primaryDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource.replica")
    public DataSource replicaDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @Primary
    public DataSource routingDataSource(
            @Qualifier("primaryDataSource") DataSource primary,
            @Qualifier("replicaDataSource") DataSource replica) {

        Map<Object, Object> dataSourceMap = new HashMap<>();
        dataSourceMap.put(DataSourceType.PRIMARY, primary);
        dataSourceMap.put(DataSourceType.REPLICA, replica);

        RoutingDataSource routingDataSource = new RoutingDataSource();
        routingDataSource.setTargetDataSources(dataSourceMap);
        routingDataSource.setDefaultTargetDataSource(primary);

        return routingDataSource;
    }

    @Bean
    public DataSource dataSource(@Qualifier("routingDataSource") DataSource routingDataSource) {
        // LazyConnectionDataSourceProxy를 감싸야 트랜잭션 시작 시점에 라우팅이 정확히 동작함
        return new LazyConnectionDataSourceProxy(routingDataSource);
    }
}
```

> **왜 LazyConnectionDataSourceProxy가 필요한가?**
> Spring은 트랜잭션 시작 시 즉시 Connection을 가져오는데, 이 시점에 `@Transactional(readOnly = true)` 속성이 아직 반영되지 않을 수 있다. `LazyConnectionDataSourceProxy`를 사용하면 실제 쿼리 실행 시점까지 Connection 획득을 지연시켜 라우팅이 정확히 동작한다.

### 3. 서비스 레이어 사용

```java
@Service
@RequiredArgsConstructor
public class UserService {

    private final UserRepository userRepository;

    // readOnly = true → Replica로 라우팅
    @Transactional(readOnly = true)
    public User findById(Long id) {
        return userRepository.findById(id)
                .orElseThrow(() -> new EntityNotFoundException("User not found: " + id));
    }

    // readOnly = false (기본값) → Primary로 라우팅
    @Transactional
    public User createUser(CreateUserRequest request) {
        User user = User.builder()
                .name(request.getName())
                .email(request.getEmail())
                .build();
        return userRepository.save(user);
    }
}
```

### 4. 다중 Replica 로드밸런싱

Replica가 여러 대라면 라운드로빈 방식으로 분산할 수 있다.

```java
public class ReplicaLoadBalancer {

    private final List<DataSource> replicas;
    private final AtomicInteger counter = new AtomicInteger(0);

    public ReplicaLoadBalancer(List<DataSource> replicas) {
        this.replicas = replicas;
    }

    public DataSource getNextReplica() {
        int index = Math.abs(counter.getAndIncrement() % replicas.size());
        return replicas.get(index);
    }
}
```

```java
// RoutingDataSource 수정 (다중 Replica 지원)
@Override
protected Object determineCurrentLookupKey() {
    if (TransactionSynchronizationManager.isCurrentTransactionReadOnly()) {
        // REPLICA_1, REPLICA_2 등을 순서대로 반환
        return replicaLoadBalancer.getNextReplicaKey();
    }
    return DataSourceType.PRIMARY;
}
```

## 주의사항 및 트레이드오프

### 1. Replication Lag 대응 전략

쓰기 후 즉시 읽어야 하는 시나리오(예: 회원가입 후 프로필 조회)에서는 반드시 Primary에서 읽어야 한다.

```java
// 쓰기 직후 조회가 필요한 경우 — readOnly = false로 Primary 강제 사용
@Transactional  // readOnly 제거 → Primary 라우팅
public UserResponse createAndFetch(CreateUserRequest request) {
    User user = createUser(request);
    return UserResponse.from(userRepository.findById(user.getId()).get());
}
```

또는 별도 어노테이션으로 강제 Primary 라우팅을 표현하는 방법도 있다.

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@Transactional(readOnly = false)
public @interface ForcePrimary {}
```

### 2. 트랜잭션 전파(Propagation) 함정

`@Transactional(readOnly = true)` 메서드 내부에서 쓰기 트랜잭션이 전파되는 경우 주의가 필요하다.

```java
// 위험한 코드
@Transactional(readOnly = true)  // Replica 연결 시작
public void riskyMethod() {
    userRepository.findAll();  // OK - Replica에서 읽기
    
    // 내부에서 REQUIRES_NEW로 쓰기 시도
    orderService.createOrder(...);  // Primary로 라우팅 되어야 하지만
                                    // 트랜잭션 전파에 따라 동작이 달라질 수 있음
}
```

**원칙**: 읽기/쓰기 트랜잭션을 명확히 분리하고, 하나의 트랜잭션에서 읽기·쓰기를 혼합하지 않는다.

### 3. DDL과 스키마 변경 시 주의

마이그레이션 도구(Flyway, Liquibase)는 반드시 Primary DataSource를 직접 지정해야 한다.

```java
@Bean
public FlywayMigrationStrategy flywayMigrationStrategy() {
    return flyway -> {
        Flyway.configure()
              .dataSource(primaryDataSource())  // Primary 명시
              .load()
              .migrate();
    };
}
```

### 4. Connection Pool 분리

`LazyConnectionDataSourceProxy`로 감싸더라도, 각 DataSource의 커넥션 풀은 독립적으로 튜닝해야 한다.

```yaml
spring:
  datasource:
    primary:
      hikari:
        maximum-pool-size: 10       # 쓰기는 상대적으로 적게
        minimum-idle: 5
    replica:
      hikari:
        maximum-pool-size: 30       # 읽기 트래픽이 많으므로 여유 있게
        minimum-idle: 10
        read-only: true             # Replica pool은 read-only 명시
```

### 5. 모니터링 지표

운영 환경에서 반드시 모니터링해야 할 지표들이다.

| 지표 | 확인 방법 (MySQL) | 임계값 기준 |
|------|-----------------|-------------|
| Replication Lag | `SHOW SLAVE STATUS` → `Seconds_Behind_Master` | 서비스별 SLA 기준 |
| Replica I/O 상태 | `Slave_IO_Running` | Must be `Yes` |
| Replica SQL 상태 | `Slave_SQL_Running` | Must be `Yes` |
| 미반영 이벤트 수 | `Relay_Log_Space` | 급격한 증가 주의 |

## 정리

데이터베이스 레플리케이션과 읽기 분산은 서비스 스케일아웃의 기본기다. 핵심을 다시 정리하면 다음과 같다.

- **`@Transactional(readOnly = true)`** 와 `AbstractRoutingDataSource`를 조합하면 코드 변경 최소화로 읽기/쓰기 분기가 가능하다.
- **`LazyConnectionDataSourceProxy`** 는 선택이 아닌 필수다. 없으면 라우팅 타이밍 이슈가 생긴다.
- **Replication Lag** 는 비동기 복제의 숙명이다. 쓰기 직후 읽기가 필요한 흐름은 설계 단계에서 명확히 식별하고, Primary 읽기를 강제하거나 캐시 전략을 병행해야 한다.
- **Connection Pool 크기** 는 Primary와 Replica를 다르게 설계한다. 일반적으로 Replica 풀이 더 커야 한다.
- 모니터링 없는 레플리케이션 운영은 언제 터질지 모르는 시한폭탄이다. **Seconds_Behind_Master** 를 항상 주시하라.

레플리케이션은 단순히 DB를 복제하는 기술이 아니라, **데이터 일관성과 성능 사이의 트레이드오프를 관리하는 아키텍처 전략**이다. 서비스의 데이터 일관성 요구사항을 정확히 파악하고, 그에 맞는 복제 전략과 라우팅 규칙을 설계하는 것이 핵심이다.
