# Web3.0과 탈중앙화 애플리케이션(dApp) 아키텍처

## 개요

Web2.0 시대의 인터넷은 구글, 아마존, 메타 같은 중앙화된 플랫폼이 데이터를 소유하고 통제하는 구조였습니다. Web3.0은 이 패러다임을 근본적으로 뒤집으려는 시도입니다. 블록체인을 기반으로 한 탈중앙화 애플리케이션, 즉 **dApp(Decentralized Application)** 은 서버가 없는 백엔드, 검열 불가능한 스토리지, 사용자 주권 데이터를 지향합니다.

백엔드 개발자 관점에서 dApp 아키텍처는 기존 Spring Boot + RDB 조합과는 전혀 다른 사고방식을 요구합니다. 스마트 컨트랙트가 비즈니스 로직을 담당하고, 이벤트 기반 데이터 동기화가 인덱싱을 대체하며, 지갑이 인증 시스템을 대신합니다. 이 포스팅에서는 dApp의 핵심 아키텍처 구성 요소를 분석하고, 실무에서 바로 적용 가능한 코드 예제를 통해 Web3 백엔드 개발의 실체를 파악해 보겠습니다.

## 핵심 개념

### dApp 아키텍처의 3계층 구조

전통적인 3-tier 아키텍처(Frontend - Backend - Database)와 비교하면 dApp은 다음과 같이 대응됩니다.

| 전통 아키텍처 | dApp 아키텍처 |
|---|---|
| Frontend (React, Vue) | Frontend + Web3 Provider |
| Backend (Spring Boot, Node.js) | Smart Contract (Solidity) |
| Database (MySQL, PostgreSQL) | On-chain Storage + IPFS |
| 인증 (JWT, Session) | 지갑 서명 (ECDSA) |
| 이벤트 큐 (Kafka) | 블록체인 이벤트 + The Graph |

### 핵심 구성 요소

**1. 스마트 컨트랙트 (Smart Contract)**

비즈니스 로직의 최소 단위입니다. EVM(Ethereum Virtual Machine) 위에서 실행되며, 한 번 배포되면 수정이 불가능합니다(업그레이드 패턴으로 우회 가능). 가스(Gas) 비용이 발생하므로 불필요한 연산은 철저히 제거해야 합니다.

**2. 노드 프로바이더 (Node Provider)**

블록체인 네트워크와 통신하는 진입점입니다. 자체 노드를 운영하거나 Infura, Alchemy, QuickNode 같은 서비스를 사용합니다. JSON-RPC 프로토콜로 통신하며, WebSocket을 통한 실시간 이벤트 구독도 지원합니다.

**3. The Graph (인덱싱 레이어)**

블록체인 데이터를 GraphQL로 쿼리할 수 있게 해주는 탈중앙화 인덱싱 프로토콜입니다. 온체인 이벤트를 구독해 Subgraph에 정의된 스키마대로 데이터를 인덱싱합니다. "블록체인의 Google"이라고 부르기도 합니다.

**4. IPFS / Arweave (탈중앙화 스토리지)**

NFT 메타데이터, 파일, 대용량 데이터를 저장하는 데 사용됩니다. 컨트랙트에는 CID(Content Identifier)만 저장해 가스 비용을 최소화합니다.

## 실전 예제

### 1. 스마트 컨트랙트 작성 (Solidity)

간단한 Todo 리스트 컨트랙트를 예시로 살펴보겠습니다.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract TodoList {
    struct Todo {
        uint256 id;
        address owner;
        string content;
        bool completed;
        uint256 createdAt;
    }

    uint256 private _todoCounter;
    
    // owner -> todoId -> Todo
    mapping(address => mapping(uint256 => Todo)) private _todos;
    mapping(address => uint256[]) private _ownerTodoIds;

    event TodoCreated(address indexed owner, uint256 indexed id, string content);
    event TodoCompleted(address indexed owner, uint256 indexed id);
    event TodoDeleted(address indexed owner, uint256 indexed id);

    modifier onlyTodoOwner(uint256 todoId) {
        require(_todos[msg.sender][todoId].owner == msg.sender, "Not the owner");
        _;
    }

    function createTodo(string calldata content) external returns (uint256) {
        require(bytes(content).length > 0, "Content cannot be empty");
        require(bytes(content).length <= 500, "Content too long");
        
        uint256 newId = ++_todoCounter;
        _todos[msg.sender][newId] = Todo({
            id: newId,
            owner: msg.sender,
            content: content,
            completed: false,
            createdAt: block.timestamp
        });
        _ownerTodoIds[msg.sender].push(newId);
        
        emit TodoCreated(msg.sender, newId, content);
        return newId;
    }

    function completeTodo(uint256 todoId) external onlyTodoOwner(todoId) {
        _todos[msg.sender][todoId].completed = true;
        emit TodoCompleted(msg.sender, todoId);
    }

    function getTodos(address owner) external view returns (Todo[] memory) {
        uint256[] memory ids = _ownerTodoIds[owner];
        Todo[] memory todos = new Todo[](ids.length);
        for (uint256 i = 0; i < ids.length; i++) {
            todos[i] = _todos[owner][ids[i]];
        }
        return todos;
    }
}
```

> **주목할 점:** `calldata` vs `memory` 사용, `indexed` 이벤트 파라미터 설계, 가스 최적화를 위한 `view` 함수 분리

### 2. Java(Web3j)로 컨트랙트 연동하기

Spring Boot 환경에서 Web3j를 사용해 컨트랙트와 상호작용하는 서비스 레이어입니다.

```java
// build.gradle
// implementation 'org.web3j:core:4.10.3'

@Service
@Slf4j
public class TodoContractService {

    private final Web3j web3j;
    private final Credentials credentials;
    private final String contractAddress;
    private final TodoList todoListContract;

    public TodoContractService(
            @Value("${web3.rpc-url}") String rpcUrl,
            @Value("${web3.private-key}") String privateKey,
            @Value("${web3.contract-address}") String contractAddress
    ) throws Exception {
        this.web3j = Web3j.build(new HttpService(rpcUrl));
        this.credentials = Credentials.create(privateKey);
        this.contractAddress = contractAddress;
        
        // 가스 가격 전략: EIP-1559 기반 동적 설정
        ContractGasProvider gasProvider = new EIP1559GasProvider(web3j, 
            ChainIdLong.MAINNET);
        
        this.todoListContract = TodoList.load(
            contractAddress, web3j, credentials, gasProvider
        );
    }

    public String createTodo(String content) throws Exception {
        TransactionReceipt receipt = todoListContract.createTodo(content).send();
        
        // 이벤트 파싱
        List<TodoList.TodoCreatedEventResponse> events = 
            todoListContract.getTodoCreatedEvents(receipt);
        
        if (events.isEmpty()) {
            throw new RuntimeException("TodoCreated event not found");
        }
        
        BigInteger todoId = events.get(0).id;
        log.info("Todo created with ID: {}, txHash: {}", todoId, receipt.getTransactionHash());
        
        return receipt.getTransactionHash();
    }

    // 읽기 전용 호출 (가스 불필요)
    public List<TodoDto> getTodos(String ownerAddress) throws Exception {
        List<TodoList.Todo> todos = todoListContract.getTodos(ownerAddress).send();
        
        return todos.stream()
            .map(todo -> TodoDto.builder()
                .id(todo.id.longValue())
                .content(todo.content)
                .completed(todo.completed)
                .createdAt(Instant.ofEpochSecond(todo.createdAt.longValue()))
                .build())
            .collect(Collectors.toList());
    }

    // 이벤트 실시간 구독
    public Disposable subscribeToTodoCreated(Consumer<TodoCreatedEvent> handler) {
        return todoListContract.todoCreatedEventFlowable(
            DefaultBlockParameterName.LATEST, DefaultBlockParameterName.LATEST
        ).subscribe(event -> {
            log.info("New todo event: owner={}, id={}", event.owner, event.id);
            handler.accept(mapToEvent(event));
        }, error -> log.error("Event subscription error", error));
    }
}
```

### 3. The Graph Subgraph 정의

온체인 이벤트를 인덱싱하는 Subgraph 스키마와 매핑 코드입니다.

```graphql
# schema.graphql
type Todo @entity {
  id: ID!
  owner: Bytes!
  content: String!
  completed: Boolean!
  createdAt: BigInt!
  completedAt: BigInt
}

type User @entity {
  id: ID!  # address
  totalTodos: BigInt!
  completedTodos: BigInt!
  todos: [Todo!]! @derivedFrom(field: "owner")
}
```

```typescript
// src/mapping.ts (AssemblyScript)
import { BigInt, Bytes } from "@graphprotocol/graph-ts";
import { TodoCreated, TodoCompleted } from "../generated/TodoList/TodoList";
import { Todo, User } from "../generated/schema";

export function handleTodoCreated(event: TodoCreated): void {
  let todoId = event.params.id.toString();
  let ownerAddress = event.params.owner.toHexString();

  // User 엔티티 upsert
  let user = User.load(ownerAddress);
  if (user === null) {
    user = new User(ownerAddress);
    user.totalTodos = BigInt.fromI32(0);
    user.completedTodos = BigInt.fromI32(0);
  }
  user.totalTodos = user.totalTodos.plus(BigInt.fromI32(1));
  user.save();

  // Todo 엔티티 생성
  let todo = new Todo(todoId);
  todo.owner = event.params.owner;
  todo.content = event.params.content;
  todo.completed = false;
  todo.createdAt = event.block.timestamp;
  todo.save();
}

export function handleTodoCompleted(event: TodoCompleted): void {
  let todo = Todo.load(event.params.id.toString());
  if (todo !== null) {
    todo.completed = true;
    todo.completedAt = event.block.timestamp;
    todo.save();
  }
}
```

### 4. 지갑 서명 기반 인증 (SIWE)

"Sign-In with Ethereum"은 Web3 표준 인증 방식입니다.

```java
@RestController
@RequestMapping("/api/auth")
public class Web3AuthController {

    @PostMapping("/nonce")
    public ResponseEntity<NonceResponse> getNonce(@RequestParam String address) {
        String nonce = generateSecureNonce();
        // Redis에 TTL 5분으로 저장
        redisTemplate.opsForValue().set("nonce:" + address, nonce, 5, TimeUnit.MINUTES);
        return ResponseEntity.ok(new NonceResponse(nonce));
    }

    @PostMapping("/verify")
    public ResponseEntity<AuthResponse> verifySignature(@RequestBody SignatureRequest req) {
        String storedNonce = redisTemplate.opsForValue()
            .getAndDelete("nonce:" + req.getAddress());
        
        if (storedNonce == null) throw new UnauthorizedException("Nonce expired");

        // EIP-4361 메시지 재구성
        String message = buildSiweMessage(req.getAddress(), storedNonce);
        
        // ECDSA 서명 검증
        String recoveredAddress = recoverAddress(message, req.getSignature());
        
        if (!recoveredAddress.equalsIgnoreCase(req.getAddress())) {
            throw new UnauthorizedException("Signature verification failed");
        }
        
        // 검증 성공 시 JWT 발급
        String jwt = jwtService.generateToken(req.getAddress());
        return ResponseEntity.ok(new AuthResponse(jwt));
    }

    private String recoverAddress(String message, String signature) {
        byte[] messageHash = Sign.getEthereumMessageHash(message.getBytes());
        Sign.SignatureData signatureData = parseSignature(signature);
        BigInteger publicKey = Sign.signedMessageHashToKey(messageHash, signatureData);
        return "0x" + Keys.getAddress(publicKey);
    }
}
```

## 주의사항 및 트레이드오프

### 불변성의 양날의 검

스마트 컨트랙트의 불변성은 신뢰의 근원이지만, 버그 수정을 불가능하게 만듭니다. **Proxy 업그레이드 패턴(OpenZeppelin의 TransparentProxy, UUPS)**을 처음부터 설계에 포함하거나, 또는 의도적으로 단순한 컨트랙트를 유지해 버그 표면적을 줄이는 전략을 선택해야 합니다.

### 온체인 데이터 비용

모든 상태 변경은 가스 비용을 수반합니다. `string` 타입의 대량 저장은 치명적으로 비쌉니다. 해시만 온체인에 저장하고 원본 데이터는 IPFS에 두는 **하이브리드 아키텍처**가 현실적인 대안입니다.

### 트랜잭션 최종성(Finality) 문제

블록체인에서 트랜잭션이 포함됐다고 즉시 확정된 것이 아닙니다. 이더리움은 약 12~15블록(약 3분) 이후를 안전한 확정으로 봅니다. 송금, 권한 변경 같은 중요 작업에서는 **충분한 블록 컨펌 수를 기다리는 로직**이 필수입니다.

### 중앙화 위험의 역설

많은 dApp이 Infura나 Alchemy 같은 중앙화 프로바이더에 의존합니다. 2022년 Infura 장애 때 MetaMask가 동작 불능 상태가 된 사례가 대표적입니다. 프로덕션 환경에서는 **다중 노드 프로바이더 Fallback 전략**과 자체 노드 운영을 병행 검토해야 합니다.

```java
// 멀티 프로바이더 Fallback 예시
public Web3j buildResilientWeb3j(List<String> rpcUrls) {
    List<Web3jService> services = rpcUrls.stream()
        .map(url -> url.startsWith("ws") ? 
            new WebSocketService(url, false) : new HttpService(url))
        .collect(Collectors.toList());
    
    return Web3j.build(new FailoverService(services));
}
```

### 개인키 관리

백엔드 서버가 개인키를 직접 보유하는 것은 심각한 보안 위협입니다. AWS KMS, HashiCorp Vault, 또는 **MPC(Multi-Party Computation) 기반 키 관리 솔루션**을 고려해야 합니다.

## 정리

Web3 dApp 아키텍처는 기존 백엔드 개발과는 근본적으로 다른 트레이드오프를 강요합니다.

| 항목 | Web2 | Web3 dApp |
|---|---|---|
| 배포 후 수정 | 언제든 가능 | 불변 (업그레이드 패턴 필요) |
| 연산 비용 | 서버 비용 | 가스 비용 (사용자 부담) |
| 데이터 쿼리 | SQL | The Graph (GraphQL) |
| 인증 | ID/PW, OAuth | 지갑 서명 (SIWE) |
| 가용성 | SLA 기반 | 네트워크 전체 가용성 |

실무에서는 **모든 것을 탈중앙화하려는 유혹을 경계**해야 합니다. 금융 로직, 소유권 증명처럼 신뢰가 핵심인 부분은 온체인에, 사용자 프로필, 설정, 캐시 레이어는 전통적인 백엔드에 두는 **하이브리드 접근**이 현실적인 선택인 경우가 많습니다.

Web3는 아직 인프라가 성숙하는 단계입니다. 도구 체인, 표준, 개발자 경험 모두 빠르게 발전하고 있으므로, 특정 라이브러리나 프로바이더에 과도하게 의존하지 않는 **추상화 레이어 설계**를 처음부터 고민하는 것이 장기적으로 유지보수 비용을 절감하는 핵심 전략입니다.
